# 麒麟 V10 Docker 版 Jenkins 自动化部署教程

> 本文由内部 Jenkins 部署记录脱敏整理而来。原文中的服务器 IP、业务项目名、仓库地址、凭据 ID、私钥路径、部署目录等信息均已替换为占位符。

## 1. 环境基准

| 项目 | 说明 |
|---|---|
| 操作系统 | 麒麟 V10 |
| 部署方式 | Docker Compose 部署 Jenkins |
| Jenkins 运行 JDK | JDK 21 |
| 后端构建 JDK | JDK 17 |
| 后端构建工具 | Maven |
| 前端构建工具 | Node.js + pnpm |
| 发布方式 | Jenkins 通过 SSH 上传产物并执行远程脚本 |
| 适用范围 | 开发、测试、演示环境 |

生产环境需要额外补充高可用、备份恢复、权限收敛、审计、制品留存、发布审批和监控告警。

## 2. 整体目标

将人工发布流程：

```text
手动拉代码
手动构建前端或后端
手动上传产物到服务器
手动重启服务或 reload nginx
```

改造成 Jenkins 流水线：

```text
GitLab 提交代码
  ↓
Jenkins 选择分支和发布环境
  ↓
Jenkins 拉取代码
  ↓
后端执行 Maven 构建 / 前端执行 pnpm 构建
  ↓
归档构建产物
  ↓
按需通过 SSH 发布到目标服务器
  ↓
远程执行服务重启、nginx reload 或状态检查
```

建议第一阶段先使用 `DEPLOY_ENV=none` 只构建不发布，确认构建稳定后再开放开发环境发布，最后再扩展测试环境。

## 3. 目录规划

Jenkins 服务器目录建议统一放在一个独立目录下：

```bash
/data/docker-jenkins
/data/docker-jenkins/docker-compose.yml
/data/docker-jenkins/Dockerfile
/data/docker-jenkins/jenkins_home
/data/docker-jenkins/cache/maven
/data/docker-jenkins/cache/pnpm
/data/docker-jenkins/artifacts
```

目录说明：

| 路径 | 说明 |
|---|---|
| `/data/docker-jenkins/docker-compose.yml` | Jenkins Docker Compose 配置 |
| `/data/docker-jenkins/Dockerfile` | 可选，自定义 Jenkins 镜像 |
| `/data/docker-jenkins/jenkins_home` | Jenkins 主数据目录，必须备份 |
| `/data/docker-jenkins/cache/maven` | Maven settings.xml 和本地仓库缓存 |
| `/data/docker-jenkins/cache/pnpm` | pnpm store 缓存 |
| `/data/docker-jenkins/artifacts` | 构建产物暂存目录 |

创建目录：

```bash
mkdir -p /data/docker-jenkins/{jenkins_home,cache/maven,cache/pnpm,artifacts}

chown -R 1000:1000 /data/docker-jenkins/jenkins_home
chown -R 1000:1000 /data/docker-jenkins/cache
chown -R 1000:1000 /data/docker-jenkins/artifacts
```

官方 `jenkins/jenkins` 镜像中的 `jenkins` 用户 UID/GID 通常为 `1000`。

## 4. Docker Compose 部署 Jenkins

### 4.1 简化版 docker-compose.yml

如果构建工具准备放到 Jenkins 全局工具或手动挂载目录中，可以先使用官方镜像：

```yaml
version: '2.4'

services:
  jenkins:
    image: jenkins/jenkins:lts-jdk21
    container_name: ci-jenkins
    restart: unless-stopped
    ports:
      - "<JENKINS_WEB_PORT>:8080"
      - "50000:50000"
    environment:
      TZ: Asia/Shanghai
      JAVA_OPTS: "-Duser.timezone=Asia/Shanghai -Dfile.encoding=UTF-8"
    volumes:
      - /data/docker-jenkins/jenkins_home:/var/jenkins_home
      - /data/docker-jenkins/cache/maven:/var/jenkins_home/.m2
      - /data/docker-jenkins/cache/pnpm:/var/jenkins_home/.pnpm-store
      - /data/docker-jenkins/artifacts:/data/artifacts
      - /etc/localtime:/etc/localtime:ro
```

说明：

- `<JENKINS_WEB_PORT>` 替换为 Jenkins Web 访问端口，例如 `18081`。
- `50000` 是 Jenkins agent 端口，不使用 agent 时也可以不暴露。
- 麒麟 V10 上如果 Docker 和 docker-compose 版本较旧，建议保留 `version: '2.4'`，避免使用 Compose v2 才支持的新语法。

### 4.2 自定义镜像 Dockerfile

如果希望镜像内直接包含 Git、SSH、Maven、JDK17、Node.js、pnpm，可创建 Dockerfile：

```dockerfile
FROM jenkins/jenkins:lts-jdk21

USER root

RUN apt-get update && apt-get install -y \
    git \
    curl \
    wget \
    unzip \
    tar \
    gzip \
    openssh-client \
    rsync \
    maven \
    openjdk-17-jdk \
    ca-certificates \
    && rm -rf /var/lib/apt/lists/*

RUN curl -fsSL https://deb.nodesource.com/setup_20.x | bash - \
    && apt-get update \
    && apt-get install -y nodejs \
    && npm install -g pnpm \
    && rm -rf /var/lib/apt/lists/*

USER jenkins
```

构建镜像：

```bash
cd /data/docker-jenkins
docker build -t local/jenkins:lts-jdk21-build .
```

然后将 Compose 中的镜像替换为：

```yaml
image: local/jenkins:lts-jdk21-build
```

## 5. 常用 Docker 命令

进入部署目录：

```bash
cd /data/docker-jenkins
```

启动：

```bash
docker-compose up -d
```

停止：

```bash
docker-compose stop
```

重启：

```bash
docker-compose restart
```

查看状态：

```bash
docker-compose ps
```

查看日志：

```bash
docker logs --tail=200 ci-jenkins
docker logs -f ci-jenkins
```

进入容器：

```bash
docker exec -it ci-jenkins bash
```

查看初始管理员密码：

```bash
docker exec ci-jenkins cat /var/jenkins_home/secrets/initialAdminPassword
```

访问地址：

```text
http://<JENKINS_SERVER_IP>:<JENKINS_WEB_PORT>
```

## 6. Jenkins 插件

推荐安装：

| 插件 | 用途 |
|---|---|
| Pipeline | 支持流水线任务 |
| Git | 拉取 Git 仓库代码 |
| GitLab | 集成 GitLab，后续可配置 Webhook |
| SSH Agent | 使用 SSH 私钥发布到服务器 |
| Git Parameter | 构建时选择 Git 分支 |
| Credentials Binding | 凭据绑定 |

注意：

- Git Parameter 的分支选择 UI 能动态拉分支，但控件样式能力有限。
- 如果使用内联 Pipeline Script，Git Parameter 有时会提示找不到 SCM。
- 更稳定的方式是把 Jenkinsfile 提交到仓库，然后 Job 改成 `Pipeline script from SCM`。

## 7. Jenkins 凭据

进入：

```text
Manage Jenkins
  -> Credentials
  -> System
  -> Global credentials
  -> Add Credentials
```

中文界面大致是：

```text
系统管理
  -> 凭据
  -> 系统
  -> 全局凭据
  -> 添加凭据
```

### 7.1 GitLab HTTP 凭据

示例：

```text
ID: <GIT_CREDENTIALS_ID>
类型: Username with password
用户名: <GITLAB_USERNAME>
密码: <GITLAB_PERSONAL_ACCESS_TOKEN>
```

Token 权限建议：

```text
read_repository
```

不建议长期使用个人登录密码。更推荐创建 Jenkins 专用 GitLab 用户，只授予必要仓库和 `read_repository` 权限。

### 7.2 发布服务器 SSH 凭据

示例：

```text
ID: <DEPLOY_CREDENTIALS_ID>
类型: SSH Username with private key
用户名: <DEPLOY_USER>
Private Key: Jenkins 发布用私钥内容
```

建议创建专用发布用户，例如 `<DEPLOY_USER>`，并只授予目标部署目录、重启脚本和 nginx reload 所需权限。开发验证阶段如果临时使用 `root`，后续应尽快收敛。

### 7.3 SSH 验证

进入 Jenkins 容器：

```bash
docker exec -it ci-jenkins bash
```

验证目标服务器连通性：

```bash
ssh -o StrictHostKeyChecking=no <DEPLOY_USER>@<TARGET_SERVER_IP> "hostname; uname -m"
```

如果使用指定私钥文件：

```bash
ssh -i /var/jenkins_home/.ssh/<DEPLOY_KEY_FILE> \
  -o StrictHostKeyChecking=no \
  <DEPLOY_USER>@<TARGET_SERVER_IP> \
  "hostname; uname -m"
```

## 8. 构建工具配置

### 8.1 Maven settings.xml

Jenkins 容器内 Maven 配置建议放到：

```bash
/var/jenkins_home/.m2/settings.xml
```

宿主机对应：

```bash
/data/docker-jenkins/cache/maven/settings.xml
```

如果使用公司 Maven 私服，`settings.xml` 可能包含账号密码，不要提交到公开仓库。文档中记录时统一使用占位符：

```xml
<server>
  <id><MAVEN_SERVER_ID></id>
  <username><MAVEN_USERNAME></username>
  <password><MAVEN_PASSWORD></password>
</server>
```

私服地址也建议脱敏：

```text
http://<MAVEN_REPOSITORY_HOST>/repository/maven-public/
```

验证：

```bash
docker exec -it ci-jenkins bash
mvn -version
ls -l /var/jenkins_home/.m2/settings.xml
```

### 8.2 Node 和 pnpm

验证：

```bash
docker exec -it ci-jenkins bash
node -v
npm -v
pnpm -v
```

建议配置 pnpm store：

```bash
pnpm config set store-dir /var/jenkins_home/.pnpm-store
```

## 9. 目标环境规划

建议统一抽象发布环境，而不是在 Jenkinsfile 中散落硬编码：

| 环境参数 | 服务器 | 后端根目录 | 前端目录 | 凭据 |
|---|---|---|---|---|
| `dev` | `<DEV_SERVER_IP>` | `<DEV_BACKEND_BASE_DIR>` | `<DEV_FRONTEND_DIST_DIR>` | `<DEV_DEPLOY_CREDENTIALS_ID>` |
| `test` | `<TEST_SERVER_IP>` | `<TEST_BACKEND_BASE_DIR>` | `<TEST_FRONTEND_DIST_DIR>` | `<TEST_DEPLOY_CREDENTIALS_ID>` |

如果存在多个目标环境，建议先只开放 `none` 和 `dev`，等开发环境跑通后再开放 `test`。

## 10. 后端 Jenkinsfile 示例

适用于 Maven 构建 Jar，并通过远程 `manage.sh` 重启服务的场景。

```groovy
pipeline {
    agent any

    options {
        timestamps()
        disableConcurrentBuilds()
        buildDiscarder(logRotator(numToKeepStr: '20'))
    }

    parameters {
        gitParameter(
            name: 'BRANCH_NAME',
            type: 'PT_BRANCH',
            branchFilter: 'origin/(.*)',
            defaultValue: 'master',
            selectedValue: 'DEFAULT',
            sortMode: 'ASCENDING_SMART',
            quickFilterEnabled: true,
            listSize: '8',
            description: '选择 Git 分支'
        )
        choice(
            name: 'DEPLOY_ENV',
            choices: ['none', 'dev', 'test'],
            description: '选择发布环境'
        )
        choice(
            name: 'SERVICE',
            choices: ['全部后端服务', '业务服务', '数据处理服务'],
            description: '选择发布范围'
        )
    }

    environment {
        GIT_CREDENTIALS_ID = '<GIT_CREDENTIALS_ID>'
        MAVEN_HOME = '/var/jenkins_home/tools/apache-maven-<MAVEN_VERSION>'
        PATH = "${env.MAVEN_HOME}/bin:${env.PATH}"
    }

    stages {
        stage('Build Info') {
            steps {
                script {
                    def deployEnv = params.DEPLOY_ENV == 'none' ? '仅构建' : params.DEPLOY_ENV
                    currentBuild.displayName = "#${env.BUILD_NUMBER} ${params.BRANCH_NAME} ${deployEnv} ${params.SERVICE}"
                    currentBuild.description = "分支：${params.BRANCH_NAME} | 环境：${deployEnv} | 服务：${params.SERVICE}"
                }
            }
        }

        stage('Checkout') {
            steps {
                git branch: params.BRANCH_NAME,
                    credentialsId: env.GIT_CREDENTIALS_ID,
                    url: '<BACKEND_GIT_REPOSITORY_URL>'
            }
        }

        stage('Build') {
            steps {
                sh '''
                    set -e
                    java -version
                    mvn -version
                    mvn clean package -DskipTests
                '''
            }
        }

        stage('Collect Artifacts') {
            steps {
                sh '''
                    set -e
                    rm -rf output
                    mkdir -p output/service-a output/service-b

                    cp <SERVICE_A_MODULE>/target/<SERVICE_A_JAR> output/service-a/<SERVICE_A_JAR>
                    cp <SERVICE_B_MODULE>/target/<SERVICE_B_JAR> output/service-b/<SERVICE_B_JAR>

                    find output -type f -name "*.jar" -print -exec ls -lh {} \\;
                '''
            }
        }

        stage('Archive') {
            steps {
                archiveArtifacts artifacts: 'output/**/*.jar', fingerprint: true
            }
        }

        stage('Deploy') {
            when {
                expression { params.DEPLOY_ENV != 'none' }
            }
            steps {
                script {
                    def targets = [
                        dev: [
                            host: '<DEV_SERVER_IP>',
                            baseDir: '<DEV_BACKEND_BASE_DIR>',
                            credentialId: '<DEV_DEPLOY_CREDENTIALS_ID>'
                        ],
                        test: [
                            host: '<TEST_SERVER_IP>',
                            baseDir: '<TEST_BACKEND_BASE_DIR>',
                            credentialId: '<TEST_DEPLOY_CREDENTIALS_ID>'
                        ]
                    ]

                    def services = [
                        '业务服务': [
                            dir: '<SERVICE_A_REMOTE_DIR>',
                            artifactDir: 'service-a',
                            jar: '<SERVICE_A_JAR>'
                        ],
                        '数据处理服务': [
                            dir: '<SERVICE_B_REMOTE_DIR>',
                            artifactDir: 'service-b',
                            jar: '<SERVICE_B_JAR>'
                        ]
                    ]

                    def target = targets[params.DEPLOY_ENV]
                    if (target == null) {
                        error "Unknown DEPLOY_ENV: ${params.DEPLOY_ENV}"
                    }

                    def serviceList = params.SERVICE == '全部后端服务'
                        ? ['业务服务', '数据处理服务']
                        : [params.SERVICE]

                    sshagent(credentials: [target.credentialId]) {
                        for (serviceName in serviceList) {
                            def svc = services[serviceName]
                            def remoteDir = "${target.baseDir}/${svc.dir}"

                            sh """
                                set -e
                                test -f output/${svc.artifactDir}/${svc.jar}
                                echo "Deploy ${serviceName} to ${target.host}:${remoteDir}"

                                scp -o StrictHostKeyChecking=no output/${svc.artifactDir}/${svc.jar} <DEPLOY_USER>@${target.host}:/tmp/${svc.jar}

                                ssh -o StrictHostKeyChecking=no <DEPLOY_USER>@${target.host} '
                                    set -e
                                    cd ${remoteDir}
                                    mkdir -p backup
                                    if [ -f ${svc.jar} ]; then
                                        cp ${svc.jar} backup/${svc.jar}.\$(date +%Y%m%d%H%M%S)
                                    fi
                                    mv /tmp/${svc.jar} ${svc.jar}
                                    chmod +x manage.sh
                                    ./manage.sh restart
                                    sleep 5
                                    ./manage.sh status
                                '
                            """
                        }
                    }
                }
            }
        }
    }
}
```

如果仓库中存在多个同名 Jar，应把 `cp` 命令改成精确模块路径，不要使用模糊 `find`。

## 11. 前端 Jenkinsfile 示例

适用于 pnpm 构建前端静态资源，并发布到 Nginx 静态目录的场景。

```groovy
pipeline {
    agent any

    options {
        timestamps()
        disableConcurrentBuilds()
        buildDiscarder(logRotator(numToKeepStr: '20'))
    }

    parameters {
        gitParameter(
            name: 'BRANCH_NAME',
            type: 'PT_BRANCH',
            branchFilter: 'origin/(.*)',
            defaultValue: 'master',
            selectedValue: 'DEFAULT',
            sortMode: 'ASCENDING_SMART',
            quickFilterEnabled: true,
            listSize: '8',
            description: '选择 Git 分支'
        )
        choice(
            name: 'DEPLOY_ENV',
            choices: ['none', 'dev', 'test'],
            description: '选择发布环境'
        )
    }

    environment {
        GIT_CREDENTIALS_ID = '<GIT_CREDENTIALS_ID>'
        NODE_HOME = '/var/jenkins_home/tools/node-v<NODE_VERSION>-linux-x64'
        PNPM_HOME = '/var/jenkins_home/.local/share/pnpm'
        PATH = "${env.NODE_HOME}/bin:${env.PNPM_HOME}:${env.PATH}"
    }

    stages {
        stage('Build Info') {
            steps {
                script {
                    def deployEnv = params.DEPLOY_ENV == 'none' ? '仅构建' : params.DEPLOY_ENV
                    currentBuild.displayName = "#${env.BUILD_NUMBER} ${params.BRANCH_NAME} ${deployEnv} 前端"
                    currentBuild.description = "分支：${params.BRANCH_NAME} | 环境：${deployEnv} | 服务：前端"
                }
            }
        }

        stage('Checkout') {
            steps {
                git branch: params.BRANCH_NAME,
                    credentialsId: env.GIT_CREDENTIALS_ID,
                    url: '<FRONTEND_GIT_REPOSITORY_URL>'
            }
        }

        stage('Install') {
            steps {
                sh '''
                    set -e
                    node -v
                    npm -v
                    corepack enable
                    corepack prepare pnpm@<PNPM_VERSION> --activate
                    pnpm -v
                    pnpm config set store-dir /var/jenkins_home/.pnpm-store
                    pnpm install --frozen-lockfile
                '''
            }
        }

        stage('Build') {
            steps {
                sh '''
                    set -e
                    pnpm run <FRONTEND_BUILD_COMMAND>
                    test -d <FRONTEND_DIST_DIR>
                    tar -zcf front-dist.tar.gz -C <FRONTEND_DIST_DIR> .
                    ls -lh front-dist.tar.gz
                '''
            }
        }

        stage('Archive') {
            steps {
                archiveArtifacts artifacts: 'front-dist.tar.gz', fingerprint: true
            }
        }

        stage('Deploy') {
            when {
                expression { params.DEPLOY_ENV != 'none' }
            }
            steps {
                script {
                    def targets = [
                        dev: [
                            host: '<DEV_SERVER_IP>',
                            frontendDir: '<DEV_FRONTEND_DIST_DIR>',
                            backupDir: '<DEV_FRONTEND_BACKUP_DIR>',
                            credentialId: '<DEV_DEPLOY_CREDENTIALS_ID>'
                        ],
                        test: [
                            host: '<TEST_SERVER_IP>',
                            frontendDir: '<TEST_FRONTEND_DIST_DIR>',
                            backupDir: '<TEST_FRONTEND_BACKUP_DIR>',
                            credentialId: '<TEST_DEPLOY_CREDENTIALS_ID>'
                        ]
                    ]

                    def target = targets[params.DEPLOY_ENV]
                    if (target == null) {
                        error "Unknown DEPLOY_ENV: ${params.DEPLOY_ENV}"
                    }

                    sshagent(credentials: [target.credentialId]) {
                        sh """
                            set -e
                            scp -o StrictHostKeyChecking=no front-dist.tar.gz <DEPLOY_USER>@${target.host}:/tmp/front-dist.tar.gz

                            ssh -o StrictHostKeyChecking=no <DEPLOY_USER>@${target.host} '
                                set -e
                                FRONT_DIR=${target.frontendDir}
                                BACKUP_DIR=${target.backupDir}
                                mkdir -p "\$FRONT_DIR" "\$BACKUP_DIR"
                                if [ -d "\$FRONT_DIR" ]; then
                                    tar -zcf "\$BACKUP_DIR/front.\$(date +%Y%m%d%H%M%S).tar.gz" -C "\$FRONT_DIR" . || true
                                fi
                                rm -rf "\$FRONT_DIR"/*
                                tar -zxf /tmp/front-dist.tar.gz -C "\$FRONT_DIR"
                                nginx -t
                                systemctl reload nginx
                            '
                        """
                    }
                }
            }
        }
    }
}
```

前端目标目录必须和 Nginx 配置中的 `root` 或 `alias` 保持一致。

## 12. Jenkins Job 配置

推荐使用两个独立 Job：

| Job | 说明 |
|---|---|
| `<BACKEND_JOB_NAME>` | 后端 Maven 构建和 Jar 发布 |
| `<FRONTEND_JOB_NAME>` | 前端 pnpm 构建和静态资源发布 |

Job 类型选择：

```text
Pipeline
```

流水线来源建议：

```text
Pipeline script from SCM
```

配置项：

```text
SCM: Git
Repository URL: <GIT_REPOSITORY_URL>
Credentials: <GIT_CREDENTIALS_ID>
Branch Specifier: */master 或 */main
Script Path: Jenkinsfile
```

如果先用内联脚本验证，也可以选择：

```text
Pipeline script
```

但后续建议把 Jenkinsfile 提交到代码仓库，便于版本管理和审计。

## 13. GitLab Webhook

进入 GitLab 项目：

```text
Settings
  -> Webhooks
```

URL 示例：

```text
http://<JENKINS_SERVER_IP>:<JENKINS_WEB_PORT>/project/<JENKINS_JOB_NAME>
```

触发事件可按需选择：

```text
Push events
Merge request events

```

开发阶段不建议一开始就启用自动发布。可以先只触发构建，发布仍保留人工参数选择。

## 14. 推荐验证顺序

1. Jenkins 页面可以正常访问。
2. Jenkins 插件安装完成。
3. GitLab 凭据可以拉取后端代码。
4. 后端 `DEPLOY_ENV=none` 构建成功。
5. GitLab 凭据可以拉取前端代码。
6. 前端 `DEPLOY_ENV=none` 构建成功。
7. Jenkins 容器可以 SSH 到开发服务器。
8. 发布前端到开发环境。
9. 发布后端到开发环境。
10. 开发环境稳定后，再启用测试环境发布。

## 15. 回滚方案

### 15.1 后端手工回滚

进入目标服务目录：

```bash
cd <BACKEND_SERVICE_DIR>
ls -lh backup
```

停止服务或准备替换：

```bash
./manage.sh stop
```

恢复备份 Jar：

```bash
cp backup/<SERVICE_JAR>.<TIMESTAMP> <SERVICE_JAR>
```

启动并检查：

```bash
./manage.sh start
./manage.sh status
```

### 15.2 前端手工回滚

进入前端目录：

```bash
cd <FRONTEND_DIST_DIR>
```

清空当前文件并恢复备份：

```bash
rm -rf ./*
tar -zxf <FRONTEND_BACKUP_DIR>/front.<TIMESTAMP>.tar.gz -C .
```

检查并重载 Nginx：

```bash
nginx -t
systemctl reload nginx
```

### 15.3 后续 Jenkins 回滚任务

后续可以新增独立回滚 Job：

| 参数 | 说明 |
|---|---|
| `DEPLOY_ENV` | 选择回滚环境 |
| `SERVICE` | 选择后端服务或前端 |
| `BACKUP_FILE` | 选择备份文件 |

动态读取远程服务器备份列表需要额外 SSH/Groovy 逻辑。初期可以先手工填写备份文件名，避免回滚任务过度复杂。

## 16. Jenkins 备份与恢复

### 16.1 备份

Jenkins 主要数据都在：

```bash
/data/docker-jenkins/jenkins_home
```

停止 Jenkins 后备份：

```bash
cd /data/docker-jenkins

```

停止 Jenkins：

```bash
cd /data/docker-jenkins
```


## 17. 常见问题

### 17.1 GitLab 拉代码失败：HTTP Basic Access denied

常见原因：

- Jenkins 未配置 GitLab 凭据。
- 凭据 ID 与 Jenkinsfile 中的 `credentialsId` 不一致。
- Token 没有 `read_repository` 权限。
- GitLab 禁止使用普通账号密码拉代码。

处理：

1. 新增 `Username with password` 凭据。
2. 密码填写 GitLab Personal Access Token。
3. Jenkinsfile 中 `credentialsId` 指向正确凭据 ID。
4. 使用最小权限 Token。

### 17.2 Maven 找不到 parent POM

常见原因：

- Jenkins 容器内没有正确的 `settings.xml`。
- 公司私服地址不可访问。
- 私服账号密码不正确。
- 本地开发机依赖了本地仓库缓存，Jenkins 中没有。

处理：

```bash
docker exec -it ci-jenkins bash
ls -l /var/jenkins_home/.m2/settings.xml
mvn -version
```

检查 `settings.xml` 中是否包含正确的私服 mirror、server 和 profile。

### 17.3 Git Parameter 页面提示没有 SCM

如果 Job 使用内联 Pipeline Script，Git Parameter 插件可能无法稳定识别 SCM。

处理方式：

- Job 改为 `Pipeline script from SCM`。
- Jenkinsfile 提交到 Git 仓库。
- 或临时把分支参数改成普通字符串参数。

### 17.4 SSH 发布失败

常见报错：

```text
Permission denied (publickey,password)
```

检查：

1. Jenkins 凭据中私钥是否完整，包括 `BEGIN` 和 `END` 行。
2. 目标服务器 `authorized_keys` 是否包含对应公钥。
3. 用户名是否正确。
4. 目标服务器 SSH 端口、防火墙、安全策略是否允许访问。
5. Jenkins 容器内是否能直接执行 SSH 验证命令。

### 17.5 manage.sh 执行失败

检查：

```bash
cd <BACKEND_SERVICE_DIR>
ls -l manage.sh
chmod +x manage.sh
./manage.sh status
```

如果 `manage.sh` 依赖父目录脚本或固定相对路径，Jenkins 远程命令必须先 `cd` 到正确目录。

### 17.6 前端发布后页面没有变化

检查：

1. Jenkins 发布目录是否是 Nginx 实际配置目录。
2. 浏览器是否缓存旧静态资源。
3. Nginx 是否 reload 成功。
4. 构建产物目录是否正确，例如 `dist`、`apps/web/dist`。
5. 前端路由 base、接口地址等环境变量是否正确。

## 18. 修改服务器配置的约束

上线自动发布前，建议明确以下边界：

- Jenkins 只负责构建和发布，不随意改业务服务器中间件配置。
- 后端服务重启优先复用现有 `manage.sh`、systemd 或 supervisor。
- 前端发布目录必须和 Nginx 配置保持一致。
- SSH 发布用户尽量使用专用账号，不长期使用 root。
- 所有凭据只放 Jenkins Credentials，不写入 Jenkinsfile。
- Jenkinsfile 中只保留占位符、凭据 ID、目录变量，不保留明文密码、Token、私钥。
