# 物联感知平台 Node-RED Demo 运行教程

本文用于从零启动并试跑一个物联感知平台 Node-RED Demo。它偏操作教程，重点是“怎么跑起来、怎么演示、怎么验证、怎么排障”。

如果你想先理解 Node-RED 的概念、节点、`msg` 生命周期、规则编排模型和平台边界，先看：[Node-RED 规则编排学习笔记](../../学习记录/Node-RED/1.Node-RED规则编排学习笔记.md)。

Demo 仓库：

```text
https://github.com/lylyuanliang/iot-platform-nodered-demo
```

## 1. Demo 是什么

这个 Demo 用 Node.js/Express 提供平台 API，用 React/Vite 提供 Web 页面，用 MySQL 存储产品、设备、遥测、告警、规则日志和指令，用 Kafka 承载事件流，并用 Node-RED 展示规则编排与流程执行。

核心闭环：

```text
设备模拟上报
  -> HTTP / MQTT 接入
  -> Kafka raw topic
  -> Node-RED 解析、标准化、规则判断
  -> 平台 API 写入 MySQL
  -> Web 页面查看告警、规则命中、指令和回执
```

逻辑上分为三端：

| 端 | Demo 对应 | 说明 |
| --- | --- | --- |
| 设备模拟端 | `#/device/*` | 模拟设备上报、拉取指令、提交回执 |
| 物联网平台端 | `#/platform/*`、server、Kafka、MySQL、Node-RED | 平台 API、事件流、规则编排、状态持久化和页面查询 |
| 业务模拟端 | `#/business/alarms` | 模拟业务系统查看告警、人工确认、触发停机或语音指令 |

## 2. 前置条件

本机需要准备：

1. Node.js 和 pnpm。
2. Docker Desktop 或可用 Docker Engine。
3. MySQL，示例中容器名为 `mysql`，账号密码为 `root/root`。
4. Kafka，示例中容器名为 `kafka-kraft`。
5. EMQX，用作 MQTT Broker。
6. Node-RED，由 demo 项目内 pnpm 启动，便于加载 `flows.json` 和 KafkaJS 节点。

检查容器：

## 5. 创建 Kafka Topic

```powershell
$topics = @(
  "iot.telemetry.raw",
  "iot.telemetry.normalized",
  "iot.invalid.data",
  "iot.rule.matched",
  "iot.alarm.created",
  "iot.command.downlink",
  "iot.command.ack"
)

foreach ($topic in $topics) {
  docker exec kafka-kraft /opt/kafka/bin/kafka-topics.sh `
    --bootstrap-server localhost:9092 `
    --create --if-not-exists `
    --topic $topic `
    --partitions 1 `
    --replication-factor 1
}
```

查看 topic：

```powershell
docker exec kafka-kraft /opt/kafka/bin/kafka-topics.sh --bootstrap-server localhost:9092 --list
```

## 8. 启动 Node-RED

本 Demo 推荐使用项目内 pnpm 启动 Node-RED：

```powershell
cd "iot-platform-nodered-demo\node-red"
pnpm exec node-red --userDir .
```

打开：

```text
http://127.0.0.1:1880
```

如果没有自动加载流程：

1. 打开 Node-RED 页面右上角菜单。
2. 选择 `Import`。
3. 选择 `node-red/flows.json`。
4. 导入后点击 `Deploy`。

`node-red/flows.json` 是核心交付物。`flows_cred.json`、`.config*.json`、`*.backup` 这类本地运行状态或凭据文件不应提交到 Git。

## 10. HTTP 上报闭环演示

### 10.1 普通温度

进入 `#/device/simulator`，设置：

```text
温度：25.5
湿度：62
在线：true
```

点击上报后，预期：

1. `telemetry_raw_log` 有 raw 记录。
2. Node-RED 消费 `iot.telemetry.raw`。
3. `telemetry_event` 有 normalized 记录。
4. 不触发高温告警。

命令行也可以模拟：

```powershell
Invoke-RestMethod `
  -Uri http://127.0.0.1:3000/api/device/report `
  -Method Post `
  -ContentType "application/json" `
  -Body '{"deviceId":"TEMP-SENSOR-001","payload":{"temp":25.5,"hum":62,"online":true}}'
```

### 10.2 高温告警

设置：

```text
温度：86.5
湿度：62
在线：true
```

预期：

1. 生成标准化遥测。
2. 触发 `HIGH_TEMPERATURE` 告警。
3. 写入 `HIGH_TEMP_AUTO_ALARM` 规则执行日志。
4. 生成 `turnOnAlarm -> ALARM-LIGHT-001` 指令。

检查 MySQL：

```powershell
docker exec mysql mysql --default-character-set=utf8mb4 -uroot -proot -D iot_nodered_demo -e "
SELECT event_id, raw_event_id, device_id, properties_json FROM telemetry_event ORDER BY id DESC LIMIT 5;
SELECT alarm_id, alarm_type, source_rule_code, status FROM alarm_record ORDER BY id DESC LIMIT 5;
SELECT rule_code, device_id, matched, result_type FROM rule_execution_log ORDER BY id DESC LIMIT 5;
SELECT command_id, device_id, service_code, command_source, status FROM command_record ORDER BY id DESC LIMIT 5;
"
```

## 11. MQTT 上报闭环演示

MQTT 路径：

```text
设备模拟端 MQTT 页面
  -> demo-server 发布 MQTT
  -> EMQX
  -> Node-RED MQTT In
  -> Kafka iot.telemetry.raw
  -> 后续复用 HTTP 上报后的同一套标准化、规则、告警、指令链路
```

操作：

1. 打开 `http://127.0.0.1:5173/#/device/mqtt-simulator`。
2. 点击“发布普通 MQTT 上报”，确认只产生 raw/normalized，不产生高温告警。
3. 点击“发布高温 MQTT 上报”，确认触发 `HIGH_TEMPERATURE`、`turnOnAlarm`。
4. 回到 Node-RED 观察 `09-MQTT接入扩展`。

MQTT topic：

```text
devices/TEMP-SENSOR-001/property/report
```

## 12. 设备指令与回执

高温规则会自动生成：

```text
turnOnAlarm -> ALARM-LIGHT-001
```

页面操作：

1. 进入 `#/device/simulator`。
2. 在执行设备指令区域点击“轮询待执行指令”。
3. 找到 `ALARM-LIGHT-001 / turnOnAlarm`。
4. 点击“成功”。
5. 进入 `#/platform/commands` 确认指令状态变为 `success`。

命令行查询：

```powershell
Invoke-RestMethod http://127.0.0.1:3000/api/device/ALARM-LIGHT-001/commands
```

提交回执：

```powershell
Invoke-RestMethod `
  -Uri http://127.0.0.1:3000/api/device/command-ack `
  -Method Post `
  -ContentType "application/json" `
  -Body '{"commandId":"替换为实际commandId","deviceId":"ALARM-LIGHT-001","status":"success","ackPayload":{"message":"alarm light turned on"}}'
```

## 13. 业务人工处理闭环

业务人工处理用于演示“高风险动作不直接由规则自动执行，而是由业务人员确认后下发”。

推荐流程：

1. 先触发一次高温告警。
2. 打开 `#/business/alarms`。
3. 找到未关闭的高温告警，点击“确认”。
4. 点击“停机处理”或“播放语音”。
5. 平台生成业务指令：
   - `stopMachine -> CNC-001`
   - `playVoice -> VOICE-SPEAKER-001`
6. 进入设备模拟器轮询目标设备指令。
7. 提交成功、失败或超时回执。
8. 回到 `#/platform/commands` 和平台仪表盘确认状态变化。

注意：业务页面的“确认 / 停机处理 / 播放语音”不直接触发 Node-RED flow。它们调用 demo-server 业务 API，更新告警或创建设备指令。Node-RED 主要负责设备上报后的解析、规则判断、告警生成和自动联动指令。

## 14. Node-RED Flow Tab 说明

| Tab | 作用 | 触发来源 | 主要影响页面 |
| --- | --- | --- | --- |
| `09-MQTT接入扩展` | MQTT 消息写入 Kafka raw | 设备 MQTT 上报、EMQX 消息 | 数据事件、告警、指令 |
| `02-Kafka原始遥测消费` | 消费 `iot.telemetry.raw` | HTTP / MQTT 上报后 | 数据事件 |
| `03-数据解析与标准化` | 原始数据解析、校验、写 normalized | raw 消息 | 数据事件、设备状态 |
| `04-规则-阈值与无效数据` | 高温阈值规则、无效数据处理 | 标准化遥测 | 告警、规则日志、仪表盘 |
| `05-规则-连续异常窗口` | 连续 3 次高温触发严重告警 | 连续高温遥测 | 告警、仪表盘 |
| `06-规则-场景联动` | 高温且在线时自动生成 `turnOnAlarm` | 高温规则命中后 | 指令、设备模拟器 |
| `07-规则-定时离线巡检` | 每 30 秒巡检离线设备 | 定时 Inject | 告警、规则日志 |
| `10-错误捕获与执行日志` | 捕获流程异常并写日志 | 其他 flow 异常 | 规则执行记录 |

这些 Flow 的概念解释见：[Node-RED 规则编排学习笔记](../../学习记录/Node-RED/1.Node-RED规则编排学习笔记.md)。

## 15. 常见问题

### 15.1 Node-RED 没有消费 Kafka

检查 consumer group：

```powershell
docker exec kafka-kraft /opt/kafka/bin/kafka-consumer-groups.sh `
  --bootstrap-server localhost:9092 `
  --describe `
  --group nodered-telemetry-normalizer
```

再检查 Node-RED 日志中是否出现 consumer joined group 之类信息。

### 15.2 MQTT 发了但没有进入 Kafka raw

检查：

1. EMQX 是否运行。
2. Node-RED 是否连接 EMQX。
3. `09-MQTT接入扩展` 是否启用并已 Deploy。
4. MQTT topic 是否为 `devices/TEMP-SENSOR-001/property/report`。

### 15.3 MySQL 没有告警

检查：

1. 温度是否大于 80。
2. Node-RED 是否正常启动。
3. server 是否监听 `3000`。
4. `POST /api/platform/alarms` 是否可用。

### 15.4 离线巡检告警很多

`07-规则-定时离线巡检` 每 30 秒运行一次。当前 Demo 未做离线告警去重，所以长时间运行会重复产生 `DEVICE_OFFLINE` 告警。正式平台应做“同一设备未关闭离线告警去重”。

### 15.5 页面时间不是北京时间或显示 ISO 格式

重启 server、web 和 Node-RED，并在 Node-RED 页面点击 `Deploy`，确保最新 flow 生效。

## 16. 推荐试跑顺序

第一次试跑建议按这个顺序：

1. 初始化数据库。
2. 创建 Kafka topics。
3. 启动 EMQX。
4. 启动 server。
5. 启动 web。
6. 启动 Node-RED。
7. 发送 HTTP 普通温度。
8. 发送 HTTP 高温。
9. 查询 `ALARM-LIGHT-001` 指令并回执。
10. 发布 MQTT 高温。
11. 在 Web 页面查看遥测、告警、规则日志、指令。
12. 在业务告警处理页面执行人工处理。
