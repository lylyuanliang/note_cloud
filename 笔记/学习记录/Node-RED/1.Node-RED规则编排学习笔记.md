# Node-RED 规则编排学习笔记

本文是 Node-RED 学习笔记，结合物联感知平台 Demo 来理解规则编排、消息流、节点、调试和平台边界。它偏知识沉淀，不是完整启动手册。

如果你要直接把 Demo 跑起来，先看：[物联感知平台 Node-RED Demo 运行教程](../../教程/物联感知/1.物联感知平台Node-RED%20Demo运行教程.md)。

Demo 仓库：

```text
https://github.com/lylyuanliang/iot-platform-nodered-demo
```

## 1. Node-RED 是什么

Node-RED 是运行在 Node.js 上的可视化流程编排工具。它通过节点和连线处理消息，适合做设备接入、协议转换、规则判断、系统联动和自动化流程。

在当前物联感知 Demo 中，Node-RED 是平台内部的规则编排组件，不是完整平台本身。

可以这样理解：

```text
设备模拟端
  -> HTTP / MQTT 上报
  -> Kafka iot.telemetry.raw
  -> Node-RED 解析、标准化、规则判断
  -> 告警 / 指令 / 规则日志
  -> demo-server 回调落库
  -> Web 页面观察闭环
```

## 2. 在物联感知平台中的边界

Node-RED 适合表达“数据进来后，经过哪些规则和动作”。它不适合替代完整物联网平台。

| 组件 | 职责 |
| --- | --- |
| demo-server | API、MySQL 持久化、查询、业务动作入口 |
| demo-web | 页面查询、人工确认、演示操作入口 |
| Kafka | 设备事件、标准化事件、告警、指令、回执等异步消息流 |
| Node-RED | 解析、标准化、规则判断、场景联动、错误捕获 |
| EMQX | MQTT Broker，承载设备 MQTT 接入 |

本 Demo 的原则：

```text
平台沉淀设备、物模型、数据、接口和状态。
Node-RED 展示规则编排、流程执行、联动动作和错误处理。
```

## 3. 核心概念

| 概念 | 说明 |
| --- | --- |
| Node | 一个处理单元，例如 MQTT In、Kafka Consumer、Function、HTTP Request、Debug |
| Flow | 一组节点连起来形成的处理流程 |
| Tab | Node-RED 画布页，用来分组展示 flow |
| msg | 节点之间传递的消息对象 |
| `msg.payload` | 最常用的主体数据字段 |
| Function | 写 JavaScript 处理消息的节点 |
| Link In/Out | 跨 tab 传递消息 |
| Catch | 捕获节点异常 |
| Debug | 显示调试输出 |
| Context | 保存临时状态，例如连续高温计数 |

## 4. msg 生命周期

`msg` 是 Node-RED 中的一次消息容器。一个节点收到 `msg`，处理后再把它传给后续节点。

常见结构：

```json
{
  "_msgid": "message-id",
  "topic": "iot.telemetry.raw",
  "payload": {
    "deviceId": "TEMP-SENSOR-001",
    "temp": 86.5
  }
}
```

常见字段：

| 字段 | 含义 |
| --- | --- |
| `msg.payload` | 主体数据 |
| `msg.topic` | 消息主题、路由标识、MQTT topic、业务分类 |
| `msg.headers` | HTTP 头等元数据 |
| `msg.req` / `msg.res` | HTTP In 场景下的请求和响应对象 |
| `msg.error` | Catch 节点捕获异常时的错误信息 |

`msg` 的结构不是全局固定的，它由上游节点和 Function 代码共同决定。例如 MQTT In、Kafka Consumer、HTTP In 输出结构都可能不同。

## 5. 分支行为

如果一个节点的同一个输出口连了多个后续节点，Node-RED 会把消息分别送到各个分支。设计时不要依赖某个分支修改 `msg.payload` 后影响另一个兄弟分支。

推荐：

```text
转换 MQTT 原始遥测
  -> 原始遥测消息转字符串
      -> 发布原始遥测消息
      -> Debug 打印
```

不推荐：

```text
分支 A 修改 msg.payload
分支 B 期望读到分支 A 修改后的值
```

如果多个分支都需要同一个新增字段，应在分叉之前统一补充。

## 6. Function 多输出

Function 有几个输出口，`return` 数组就按位置对应几个输出口。

```javascript
return [msg, null];          // 只走第 1 个输出口
return [null, msg];          // 只走第 2 个输出口
return [msg1, msg2];         // 两个输出口都走
return null;                 // 所有输出口都不走
return [[msg1, msg2], null]; // 第 1 个输出口连续发多条消息
```

当前 Demo 中 `解析并标准化遥测数据` 是典型多输出：

| 输出口 | 含义 |
| --- | --- |
| 第 1 个输出口 | 标准化成功，进入保存、发布 Kafka、规则判断 |
| 第 2 个输出口 | 数据无效，进入无效数据保存和发布 |

如果数据无效，只会进入无效数据分支，不会同时进入标准化保存和规则判断。

## 7. 常用节点速查

### 7.1 通用节点

| 节点 | 用途 | Demo 例子 |
| --- | --- | --- |
| `inject` | 手动或定时产生消息 | 离线巡检每 30 秒触发 |
| `debug` | 打印消息 | 查看 Kafka raw 或标准化结果 |
| `catch` | 捕获节点异常 | 统一记录规则执行错误 |
| `status` | 监听节点状态 | 观察 Kafka/MQTT 是否连接 |
| `link in` / `link out` | 跨 tab 传递消息 | raw 消息送到标准化 tab |
| `comment` | 画布说明 | 标注成功分支、失败分支、平台回调 |

### 7.2 功能节点

| 节点 | 用途 | Demo 例子 |
| --- | --- | --- |
| `function` | 写 JavaScript 处理消息 | 解析遥测、判断高温、组装告警 |
| `switch` | 条件分流 | 判断温度、在线状态 |
| `change` | 修改、设置、删除字段 | 补充默认来源字段 |
| `template` | 模板生成文本或 JSON | 生成告警详情 |
| `delay` | 延迟或限速 | 模拟设备响应慢 |
| `trigger` | 收到消息后立即/延迟输出 | 模拟离线检测 |

### 7.3 网络和集成节点

| 节点 | 用途 | Demo 例子 |
| --- | --- | --- |
| `http in` | 创建 HTTP 入口 | 设备上报模拟入口 |
| `http response` | 返回 HTTP 响应 | 返回 accepted |
| `http request` | 调用外部 API | 回调 demo-server 保存告警 |
| `mqtt in` | 订阅 MQTT topic | `devices/+/property/report` |
| Kafka Consumer | 消费 Kafka topic | `iot.telemetry.raw` |
| Kafka Producer | 发布 Kafka topic | `iot.alarm.created`、`iot.command.downlink` |

## 8. 物模型：属性、事件、服务

物联感知平台中，物模型通常从属性、事件、服务三个维度描述设备。

| 维度 | 含义 | 示例 |
| --- | --- | --- |
| 属性 | 设备当前状态或测量值 | 温度、湿度、在线状态 |
| 事件 | 设备或平台产生的事件 | 高温事件、离线事件 |
| 服务 | 平台可调用的设备能力 | 打开声光报警、停止设备、播放语音 |

标准遥测事件示例：

```json
{
  "eventId": "EVT-20260715-0001",
  "productKey": "temperature-sensor",
  "deviceId": "TEMP-SENSOR-001",
  "eventType": "property_report",
  "properties": {
    "temperature": 86.5,
    "humidity": 62,
    "online": true
  },
  "source": "http"
}
```

设备服务指令示例：

```json
{
  "commandId": "CMD-20260715-0001",
  "deviceId": "ALARM-LIGHT-001",
  "serviceCode": "turnOnAlarm",
  "params": {
    "level": "warning",
    "durationSeconds": 30
  }
}
```

## 9. 规则编排模型

可以用这个模型理解场景联动：

```text
Trigger 触发源
  -> Condition 条件
  -> Action 动作
  -> Execution 执行方式
  -> Result 执行结果
```

触发源：

| 触发源 | Node-RED 实现 | Demo 示例 |
| --- | --- | --- |
| 定时触发 | Inject 定时 | 每 30 秒巡检离线设备 |
| 手动触发 | HTTP / 业务 API | 业务系统点击处理告警 |
| 设备触发 | Kafka/MQTT In | 温度上报触发规则 |

条件：

```text
temperature > 80
online == true
连续 3 次超温
设备 60 秒未上报
```

动作：

1. 生成告警。
2. 写规则命中日志。
3. 调平台 API。
4. 写 Kafka 指令 topic。
5. 下发设备服务。
6. 回写执行状态。

## 10. Demo 中的关键规则

### 10.1 阈值告警

```text
Kafka In: iot.telemetry.normalized
  -> 判断 temperature > 80
  -> 生成 HIGH_TEMPERATURE 告警
  -> 写 iot.alarm.created
  -> HTTP 回调平台保存告警
```

### 10.2 连续异常窗口

用 `flow.get()` 和 `flow.set()` 记录设备连续高温次数。

```javascript
const deviceId = msg.payload.deviceId;
const temperature = msg.payload.properties.temperature;
const key = `highTempCount:${deviceId}`;
let count = flow.get(key) || 0;

if (temperature > 80) {
  count += 1;
} else {
  count = 0;
}

flow.set(key, count);

if (count >= 3) {
  msg.payload = {
    deviceId,
    alarmType: "CONTINUOUS_HIGH_TEMPERATURE",
    alarmLevel: "critical"
  };
  return msg;
}

return null;
```

### 10.3 场景联动

```text
温度高 + 设备在线
  -> 生成 turnOnAlarm 指令
  -> 写 iot.command.downlink
  -> 设备模拟器拉取指令
  -> 设备提交回执
```

### 10.4 定时离线巡检

```text
Inject: 每 30 秒
  -> GET /api/devices
  -> 找出超过 60 秒未上报的设备
  -> 生成 DEVICE_OFFLINE 告警
```

## 12. 调试方法

推荐从最短链路开始：

```text
Inject -> Debug
HTTP In -> JSON -> Debug -> HTTP Response
HTTP In -> Function 标准化 -> Debug
标准化 -> Kafka Out
Kafka In -> Switch -> Debug
规则命中 -> 告警 -> Kafka Out
指令下发 -> 回执 -> 状态更新
```

调试建议：

1. 打开 Node-RED 右侧 Debug 面板。
2. 需要看消息时，在关键节点后接 Debug 节点。
3. 输出建议先选 `complete msg object`。
4. 修改节点后点击 `Deploy`。
5. 用 `_msgid` 或业务 `eventId` 追踪同一条消息。
6. 关键流程接 Catch，避免规则失败不可见。

## 13. 常见坑

| 现象 | 原因或处理 |
| --- | --- |
| Debug 面板没日志 | 没接 Debug 节点，或消息没经过该 Debug |
| 节点下方显示 `Idle` | 节点插件自己的状态，表示当前空闲 |
| Catch 节点没有输入线 | 正常，Catch 监听运行时异常，不走普通输入线 |
| 多条线从一个节点发出 | 同一输出口的消息会复制给多个后续节点 |
| `return [null, msg]` | 只走第 2 个输出口 |
| Kafka Producer 写对象失败 | Kafka value 通常需要字符串或 Buffer |
| Context 当数据库用 | Context 适合短期状态，核心状态仍应写 MySQL |

## 14. 技术分享讲法

建议按这个顺序讲：

1. 物联感知平台为什么需要规则编排。
2. Node-RED 是什么，以及它不是什么。
3. 产品、设备、物模型：属性、事件、服务。
4. HTTP、MQTT、Kafka 在规则编排中的同级通道定位。
5. 本次 Demo 的三端架构。
6. 数据解析、标准化和预处理。
7. 阈值告警、连续异常、定时离线巡检。
8. 自动动作与人工确认动作的区别。
9. 设备指令下发与回执闭环。
10. 错误捕获、执行日志和链路诊断。
11. `flows.json` 作为规则编排源码资产。

## 15. 适用边界

适合：

1. 可视化规则编排。
2. 协议和系统集成。
3. 轻量数据解析和路由。
4. 场景联动原型。
5. 运维可理解的自动化流程。

不适合：

1. 完整设备主数据管理。
2. 完整权限体系。
3. 强事务业务流程。
4. 大规模复杂业务代码。
5. 完整物联网平台替代品。
