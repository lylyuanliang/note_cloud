# 设计模式对比速查

设计模式之间看起来相似，通常是因为它们都在使用接口、多态、组合、委托这些基础手段。真正需要记住的是：同样的业务场景中，代码结构为什么会变成不同样子。

本文按“容易混淆的一组模式尽量使用同一个业务域”的方式对比。不要强行把 23 种模式放进同一个业务场景；那样会让部分模式变得牵强。

## 1. 总体记忆框架

| 分类 | 关注点 | 记忆句 |
| --- | --- | --- |
| 创建型模式 | 对象怎么创建 | 把 `new` 的复杂性藏起来 |
| 结构型模式 | 对象怎么组合 | 用组合、包装、代理组织对象关系 |
| 行为型模式 | 对象之间怎么协作 | 把算法、状态、请求、通知等行为拆开 |

## 2. 日志系统：简单工厂、工厂方法、抽象工厂

### 2.1 统一业务场景

系统需要记录日志。最开始只有文件日志，后来有数据库日志，再后来发现每种日志不是一个对象就能完成，而是一整套对象：日志写入器、日志格式化器、日志存储器。

### 2.2 简单工厂：一个工厂按参数判断

适合日志类型少、变化不频繁的阶段。

```java
interface LogWriter {
    void write(String message);
}

class FileLogWriter implements LogWriter {
    public void write(String message) {
        System.out.println("写入文件日志：" + message);
    }
}

class DatabaseLogWriter implements LogWriter {
    public void write(String message) {
        System.out.println("写入数据库日志：" + message);
    }
}

class SimpleLogFactory {
    public static LogWriter create(String type) {
        if ("file".equals(type)) {
            return new FileLogWriter();
        }
        if ("database".equals(type)) {
            return new DatabaseLogWriter();
        }
        throw new IllegalArgumentException("未知日志类型：" + type);
    }
}

public class Client {
    public static void main(String[] args) {
        LogWriter writer = SimpleLogFactory.create("file");
        writer.write("系统启动");
    }
}
```

代码特征：`create(type)` 里有 `if-else` 或 `switch`。

### 2.3 工厂方法：每种日志一个工厂

适合日志类型会扩展，希望新增类型时不改已有工厂逻辑。

```java
interface LogWriter {
    void write(String message);
}

class FileLogWriter implements LogWriter {
    public void write(String message) {
        System.out.println("写入文件日志：" + message);
    }
}

class DatabaseLogWriter implements LogWriter {
    public void write(String message) {
        System.out.println("写入数据库日志：" + message);
    }
}

abstract class LogFactory {
    public abstract LogWriter createWriter();
}

class FileLogFactory extends LogFactory {
    public LogWriter createWriter() {
        return new FileLogWriter();
    }
}

class DatabaseLogFactory extends LogFactory {
    public LogWriter createWriter() {
        return new DatabaseLogWriter();
    }
}

public class Client {
    public static void main(String[] args) {
        LogFactory factory = new FileLogFactory();
        LogWriter writer = factory.createWriter();
        writer.write("系统启动");
    }
}
```

代码特征：一个抽象工厂方法只创建一种产品等级结构，比如这里只创建 `LogWriter`。

### 2.4 抽象工厂：每种日志创建一整套相关对象

适合“文件日志”和“云日志”都不是单个对象，而是一整套配套对象。

```java
interface LogWriter {
    void write(String message);
}

interface LogFormatter {
    String format(String message);
}

class FileLogWriter implements LogWriter {
    public void write(String message) {
        System.out.println("文件日志：" + message);
    }
}

class FileLogFormatter implements LogFormatter {
    public String format(String message) {
        return "[FILE] " + message;
    }
}

class CloudLogWriter implements LogWriter {
    public void write(String message) {
        System.out.println("云日志：" + message);
    }
}

class CloudLogFormatter implements LogFormatter {
    public String format(String message) {
        return "{\"type\":\"cloud\",\"message\":\"" + message + "\"}";
    }
}

interface LogComponentFactory {
    LogWriter createWriter();
    LogFormatter createFormatter();
}

class FileLogComponentFactory implements LogComponentFactory {
    public LogWriter createWriter() {
        return new FileLogWriter();
    }

    public LogFormatter createFormatter() {
        return new FileLogFormatter();
    }
}

class CloudLogComponentFactory implements LogComponentFactory {
    public LogWriter createWriter() {
        return new CloudLogWriter();
    }

    public LogFormatter createFormatter() {
        return new CloudLogFormatter();
    }
}

public class Client {
    public static void main(String[] args) {
        LogComponentFactory factory = new CloudLogComponentFactory();
        LogFormatter formatter = factory.createFormatter();
        LogWriter writer = factory.createWriter();
        writer.write(formatter.format("系统启动"));
    }
}
```

代码特征：一个工厂里有多个创建方法，创建的是同一产品族的配套对象。

### 2.5 选型判断

| 业务变化 | 更适合 |
| --- | --- |
| 根据一个参数临时选择日志实现 | 简单工厂 |
| 日志写入器类型经常增加 | 工厂方法 |
| 每种日志都有一整套配套组件 | 抽象工厂 |

## 3. 报表导出：工厂、建造者、原型

### 3.1 统一业务场景

系统要导出报表。不同报表类型不同，报表配置又有很多可选项，同时还存在“复制已有报表模板后微调”的需求。

### 3.2 工厂：选择创建哪一种报表

```java
interface Report {
    void export();
}

class SalesReport implements Report {
    public void export() {
        System.out.println("导出销售报表");
    }
}

class FinanceReport implements Report {
    public void export() {
        System.out.println("导出财务报表");
    }
}

class ReportFactory {
    public static Report create(String type) {
        if ("sales".equals(type)) {
            return new SalesReport();
        }
        if ("finance".equals(type)) {
            return new FinanceReport();
        }
        throw new IllegalArgumentException("未知报表类型");
    }
}
```

重点：根据类型创建不同报表对象。

### 3.3 建造者：一步步组装复杂报表配置

```java
class ReportConfig {
    private String title;
    private String[] columns;
    private String filter;
    private String sort;

    public static class Builder {
        private final ReportConfig config = new ReportConfig();

        public Builder title(String title) {
            config.title = title;
            return this;
        }

        public Builder columns(String... columns) {
            config.columns = columns;
            return this;
        }

        public Builder filter(String filter) {
            config.filter = filter;
            return this;
        }

        public Builder sort(String sort) {
            config.sort = sort;
            return this;
        }

        public ReportConfig build() {
            return config;
        }
    }
}

ReportConfig config = new ReportConfig.Builder()
        .title("月度销售报表")
        .columns("日期", "订单数", "销售额")
        .filter("month = 2026-07")
        .sort("销售额 desc")
        .build();
```

重点：对象参数多，构建过程需要可读、可控。

### 3.4 原型：复制已有报表模板

```java
class ReportTemplate implements Cloneable {
    private String title;
    private String filter;

    public ReportTemplate(String title, String filter) {
        this.title = title;
        this.filter = filter;
    }

    public void setFilter(String filter) {
        this.filter = filter;
    }

    public ReportTemplate clone() {
        try {
            return (ReportTemplate) super.clone();
        } catch (CloneNotSupportedException e) {
            throw new AssertionError(e);
        }
    }
}

ReportTemplate monthTemplate = new ReportTemplate("月度销售报表", "month = current");
ReportTemplate julyTemplate = monthTemplate.clone();
julyTemplate.setFilter("month = 2026-07");
```

重点：已有一个模板对象，复制后只改少量字段。

### 3.5 选型判断

| 业务变化 | 更适合 |
| --- | --- |
| 报表类型不同 | 工厂 |
| 报表配置参数多 | 建造者 |
| 报表模板相似，只改少量字段 | 原型 |

## 4. 支付系统：适配器、桥接、装饰器、代理

### 4.1 统一业务场景

系统需要支付。它要接入第三方支付，要支持不同通知渠道，要给支付叠加积分，还要在支付前做风控。

统一接口如下：

```java
interface PayService {
    void pay(String orderId, int amount);
}
```

### 4.2 适配器：三方接口不兼容

```java
class ThirdPartyPay {
    public void createPayment(String tradeNo, int cents) {
        System.out.println("第三方支付：" + tradeNo + "，金额：" + cents);
    }
}

class ThirdPartyPayAdapter implements PayService {
    private final ThirdPartyPay thirdPartyPay;

    public ThirdPartyPayAdapter(ThirdPartyPay thirdPartyPay) {
        this.thirdPartyPay = thirdPartyPay;
    }

    public void pay(String orderId, int amount) {
        thirdPartyPay.createPayment(orderId, amount);
    }
}
```

重点：把第三方的 `createPayment()` 转成系统需要的 `pay()`。

### 4.3 桥接：支付方式和通知渠道独立变化

```java
interface PayNotifier {
    void notify(String orderId);
}

class SmsPayNotifier implements PayNotifier {
    public void notify(String orderId) {
        System.out.println("短信通知支付成功：" + orderId);
    }
}

abstract class AbstractPayService implements PayService {
    protected final PayNotifier notifier;

    protected AbstractPayService(PayNotifier notifier) {
        this.notifier = notifier;
    }
}

class OnlinePayService extends AbstractPayService {
    public OnlinePayService(PayNotifier notifier) {
        super(notifier);
    }

    public void pay(String orderId, int amount) {
        System.out.println("线上支付：" + orderId);
        notifier.notify(orderId);
    }
}
```

重点：支付方式和通知渠道是两个变化维度，不用继承组合出 `SmsOnlinePayService`、`EmailOnlinePayService` 等一堆类。

### 4.4 装饰器：支付后叠加积分

```java
class BasicPayService implements PayService {
    public void pay(String orderId, int amount) {
        System.out.println("支付订单：" + orderId);
    }
}

class PointPayDecorator implements PayService {
    private final PayService payService;

    public PointPayDecorator(PayService payService) {
        this.payService = payService;
    }

    public void pay(String orderId, int amount) {
        payService.pay(orderId, amount);
        System.out.println("增加积分：" + amount / 10);
    }
}
```

重点：原支付功能还在，外面再包一层增加积分。

### 4.5 代理：支付前做风控

```java
class RiskControlPayProxy implements PayService {
    private final PayService payService;

    public RiskControlPayProxy(PayService payService) {
        this.payService = payService;
    }

    public void pay(String orderId, int amount) {
        if (amount > 100000) {
            System.out.println("风控拦截：" + orderId);
            return;
        }
        payService.pay(orderId, amount);
    }
}
```

重点：代理可能阻止真实支付对象被调用。

### 4.6 选型判断

| 业务变化 | 更适合 |
| --- | --- |
| 第三方接口和系统接口不一致 | 适配器 |
| 支付方式和通知渠道都要扩展 | 桥接 |
| 支付后叠加积分、优惠券、成长值 | 装饰器 |
| 支付前做权限、风控、缓存、事务 | 代理 |

## 5. 菜单系统：组合、迭代器、访问者

### 5.1 统一业务场景

后台系统有多级菜单。菜单可以包含子菜单，也可以是具体菜单项。后续要遍历菜单，还要统计菜单数量、导出菜单。

### 5.2 组合：表达树形结构

```java
interface MenuComponent {
    void print();
}

class MenuItem implements MenuComponent {
    private final String name;

    public MenuItem(String name) {
        this.name = name;
    }

    public void print() {
        System.out.println("菜单项：" + name);
    }
}

class MenuGroup implements MenuComponent {
    private final String name;
    private final List<MenuComponent> children = new ArrayList<>();

    public MenuGroup(String name) {
        this.name = name;
    }

    public void add(MenuComponent component) {
        children.add(component);
    }

    public void print() {
        System.out.println("菜单组：" + name);
        for (MenuComponent child : children) {
            child.print();
        }
    }
}
```

重点：菜单组和菜单项都实现同一个接口，客户端可以一致处理。

### 5.3 迭代器：隐藏菜单内部遍历方式

```java
interface MenuIterator {
    boolean hasNext();
    MenuComponent next();
}

class ListMenuIterator implements MenuIterator {
    private final List<MenuComponent> menus;
    private int index = 0;

    public ListMenuIterator(List<MenuComponent> menus) {
        this.menus = menus;
    }

    public boolean hasNext() {
        return index < menus.size();
    }

    public MenuComponent next() {
        return menus.get(index++);
    }
}
```

重点：外部不关心菜单内部是 `List`、数组还是树，只通过 `hasNext()/next()` 遍历。

### 5.4 访问者：给菜单结构增加新操作

```java
interface MenuVisitor {
    void visit(MenuItem item);
    void visit(MenuGroup group);
}

interface VisitableMenu {
    void accept(MenuVisitor visitor);
}

class CountMenuVisitor implements MenuVisitor {
    private int count = 0;

    public void visit(MenuItem item) {
        count++;
    }

    public void visit(MenuGroup group) {
        count++;
    }

    public int getCount() {
        return count;
    }
}
```

重点：菜单结构稳定，但统计、导出、权限检查等操作可能持续增加。

### 5.5 选型判断

| 业务变化 | 更适合 |
| --- | --- |
| 菜单有父子层级 | 组合 |
| 只想统一遍历菜单 | 迭代器 |
| 菜单结构稳定，但要增加统计、导出等操作 | 访问者 |

## 6. 订单系统：策略、状态、模板方法

### 6.1 统一业务场景

订单系统要计算价格、处理订单流程，还要根据订单状态限制操作。这里很容易混淆策略、状态和模板方法。

### 6.2 策略：价格算法可替换

```java
interface PriceStrategy {
    int calculate(int price);
}

class NormalPriceStrategy implements PriceStrategy {
    public int calculate(int price) {
        return price;
    }
}

class VipPriceStrategy implements PriceStrategy {
    public int calculate(int price) {
        return (int) (price * 0.8);
    }
}

class OrderPriceService {
    private final PriceStrategy strategy;

    public OrderPriceService(PriceStrategy strategy) {
        this.strategy = strategy;
    }

    public int calculate(int price) {
        return strategy.calculate(price);
    }
}
```

重点：算法由外部选择，运行时可以替换。

### 6.3 状态：订单状态不同，行为不同

```java
interface OrderState {
    void pay(OrderContext order);
    void ship(OrderContext order);
}

class UnpaidState implements OrderState {
    public void pay(OrderContext order) {
        System.out.println("支付成功");
        order.setState(new PaidState());
    }

    public void ship(OrderContext order) {
        System.out.println("未支付，不能发货");
    }
}

class PaidState implements OrderState {
    public void pay(OrderContext order) {
        System.out.println("订单已支付");
    }

    public void ship(OrderContext order) {
        System.out.println("发货成功");
    }
}

class OrderContext {
    private OrderState state = new UnpaidState();

    public void setState(OrderState state) {
        this.state = state;
    }

    public void pay() {
        state.pay(this);
    }

    public void ship() {
        state.ship(this);
    }
}
```

重点：状态由订单内部流程切换，状态变了行为也变。

### 6.4 模板方法：订单处理流程固定

```java
abstract class OrderProcessor {
    public final void process() {
        validate();
        lockStock();
        pay();
        notifyUser();
    }

    protected void validate() {
        System.out.println("校验订单");
    }

    protected abstract void lockStock();

    protected abstract void pay();

    protected void notifyUser() {
        System.out.println("通知用户");
    }
}

class NormalOrderProcessor extends OrderProcessor {
    protected void lockStock() {
        System.out.println("锁定普通商品库存");
    }

    protected void pay() {
        System.out.println("普通订单支付");
    }
}
```

重点：父类固定流程顺序，子类只填某些步骤。

### 6.5 选型判断

| 业务变化 | 更适合 |
| --- | --- |
| 价格算法不同 | 策略 |
| 订单状态不同导致行为不同 | 状态 |
| 订单处理步骤固定，部分步骤变化 | 模板方法 |

## 7. 审批系统：责任链、命令、备忘录

### 7.1 统一业务场景

报销审批系统中，申请要经过不同级别的人审批；用户还可以提交、撤销、重提；编辑申请时需要保存历史版本。

### 7.2 责任链：谁能审批谁处理

```java
abstract class Approver {
    protected Approver next;

    public Approver setNext(Approver next) {
        this.next = next;
        return next;
    }

    public abstract void approve(int amount);
}

class Manager extends Approver {
    public void approve(int amount) {
        if (amount <= 1000) {
            System.out.println("经理审批通过");
        } else if (next != null) {
            next.approve(amount);
        }
    }
}

class Director extends Approver {
    public void approve(int amount) {
        if (amount <= 5000) {
            System.out.println("总监审批通过");
        } else if (next != null) {
            next.approve(amount);
        }
    }
}
```

重点：请求沿着链传递，直到有人处理。

### 7.3 命令：把操作封装成对象

```java
interface Command {
    void execute();
}

class ExpenseService {
    public void submit() {
        System.out.println("提交报销");
    }

    public void cancel() {
        System.out.println("撤销报销");
    }
}

class SubmitExpenseCommand implements Command {
    private final ExpenseService service;

    public SubmitExpenseCommand(ExpenseService service) {
        this.service = service;
    }

    public void execute() {
        service.submit();
    }
}

class CancelExpenseCommand implements Command {
    private final ExpenseService service;

    public CancelExpenseCommand(ExpenseService service) {
        this.service = service;
    }

    public void execute() {
        service.cancel();
    }
}
```

重点：提交、撤销、重提都可以作为命令排队、记录或重放。

### 7.4 备忘录：保存申请编辑历史

```java
class ExpenseDraft {
    private String content;

    public void setContent(String content) {
        this.content = content;
    }

    public ExpenseMemento save() {
        return new ExpenseMemento(content);
    }

    public void restore(ExpenseMemento memento) {
        this.content = memento.getContent();
    }
}

class ExpenseMemento {
    private final String content;

    public ExpenseMemento(String content) {
        this.content = content;
    }

    public String getContent() {
        return content;
    }
}
```

重点：保存的是对象状态，用于撤销编辑，不是封装业务操作。

### 7.5 选型判断

| 业务变化 | 更适合 |
| --- | --- |
| 不同金额由不同角色审批 | 责任链 |
| 提交、撤销、重提需要对象化 | 命令 |
| 编辑过程需要撤销恢复 | 备忘录 |

## 8. 消息系统：观察者、中介者

### 8.1 统一业务场景

系统中有消息通知。一种情况是“系统公告发布后通知所有订阅者”；另一种情况是“多个用户在聊天室里互相发消息”。

### 8.2 观察者：一个主题广播给多个观察者

```java
interface MessageObserver {
    void update(String message);
}

class AnnouncementSubject {
    private final List<MessageObserver> observers = new ArrayList<>();

    public void addObserver(MessageObserver observer) {
        observers.add(observer);
    }

    public void publish(String message) {
        for (MessageObserver observer : observers) {
            observer.update(message);
        }
    }
}
```

重点：发布者只负责广播，不协调观察者之间的关系。

### 8.3 中介者：多个对象通过中介交互

```java
interface ChatMediator {
    void send(String message, ChatUser sender);
}

class ChatRoom implements ChatMediator {
    private final List<ChatUser> users = new ArrayList<>();

    public void addUser(ChatUser user) {
        users.add(user);
    }

    public void send(String message, ChatUser sender) {
        for (ChatUser user : users) {
            if (user != sender) {
                user.receive(message);
            }
        }
    }
}

class ChatUser {
    private final ChatMediator mediator;

    public ChatUser(ChatMediator mediator) {
        this.mediator = mediator;
    }

    public void send(String message) {
        mediator.send(message, this);
    }

    public void receive(String message) {
        System.out.println("收到消息：" + message);
    }
}
```

重点：用户之间不直接引用，聊天室负责协调谁收到消息。

### 8.4 选型判断

| 业务变化 | 更适合 |
| --- | --- |
| 一个对象变化，通知多个订阅者 | 观察者 |
| 多个对象互相协作，关系复杂 | 中介者 |

## 9. 权限规则：解释器、策略

### 9.1 统一业务场景

权限系统要判断用户能不能访问资源。有时规则是一段表达式，例如 `ADMIN && AUDITOR`；有时只是不同权限算法之间切换。

### 9.2 解释器：解析权限表达式

```java
interface PermissionExpression {
    boolean interpret(Set<String> roles);
}

class RoleExpression implements PermissionExpression {
    private final String role;

    public RoleExpression(String role) {
        this.role = role;
    }

    public boolean interpret(Set<String> roles) {
        return roles.contains(role);
    }
}

class AndExpression implements PermissionExpression {
    private final PermissionExpression left;
    private final PermissionExpression right;

    public AndExpression(PermissionExpression left, PermissionExpression right) {
        this.left = left;
        this.right = right;
    }

    public boolean interpret(Set<String> roles) {
        return left.interpret(roles) && right.interpret(roles);
    }
}
```

重点：表达式会被拆成语法节点，节点之间能组合。

### 9.3 策略：切换权限判断算法

```java
interface PermissionStrategy {
    boolean allow(User user, Resource resource);
}

class RolePermissionStrategy implements PermissionStrategy {
    public boolean allow(User user, Resource resource) {
        return user.hasRole(resource.requiredRole());
    }
}

class OwnerPermissionStrategy implements PermissionStrategy {
    public boolean allow(User user, Resource resource) {
        return user.id().equals(resource.ownerId());
    }
}

class PermissionService {
    private final PermissionStrategy strategy;

    public PermissionService(PermissionStrategy strategy) {
        this.strategy = strategy;
    }

    public boolean allow(User user, Resource resource) {
        return strategy.allow(user, resource);
    }
}
```

重点：算法可替换，但不解析语法表达式。

### 9.4 选型判断

| 业务变化 | 更适合 |
| --- | --- |
| 规则是一段表达式，需要解析组合 | 解释器 |
| 只是多种权限判断算法切换 | 策略 |

## 10. 其他容易混淆的结构型模式

### 10.1 享元：地图点位图标复用

地图上有大量监控点，很多点使用同一种图标。如果每个点都创建一个完整图标对象，会浪费内存。

```java
class MarkerIcon {
    private final String type;

    public MarkerIcon(String type) {
        this.type = type;
    }

    public void draw(int x, int y) {
        System.out.println(type + " 图标绘制在：" + x + "," + y);
    }
}

class MarkerIconFactory {
    private static final Map<String, MarkerIcon> CACHE = new HashMap<>();

    public static MarkerIcon getIcon(String type) {
        return CACHE.computeIfAbsent(type, MarkerIcon::new);
    }
}
```

重点：共享内部状态 `type`，坐标 `x,y` 作为外部状态传入。

### 10.2 外观：下单流程统一入口

下单需要调用库存、价格、支付、通知多个子系统。外观模式把复杂流程包装成一个简单入口。

```java
class OrderFacade {
    private final StockService stockService = new StockService();
    private final PayService payService = new PayService();
    private final NotifyService notifyService = new NotifyService();

    public void createOrder(String orderId) {
        stockService.lock(orderId);
        payService.pay(orderId);
        notifyService.send(orderId);
        System.out.println("下单完成");
    }
}
```

重点：客户端只调用 `createOrder()`，不用知道内部子系统顺序。

### 10.3 享元和外观的区别

| 模式 | 业务目标 | 代码特征 |
| --- | --- | --- |
| 享元 | 减少大量重复对象 | 工厂里有缓存 `CACHE` |
| 外观 | 简化复杂子系统调用 | 外观类里组合多个子系统 |

## 11. 容易混淆模式速查表

| 如果你想做的是 | 优先考虑 |
| --- | --- |
| 根据类型创建对象 | 简单工厂/工厂方法 |
| 创建一整套相关对象 | 抽象工厂 |
| 分步骤构建复杂对象 | 建造者 |
| 复制已有对象 | 原型 |
| 兼容旧接口或三方接口 | 适配器 |
| 拆分两个独立变化维度 | 桥接 |
| 给对象动态加功能 | 装饰器 |
| 控制访问、加权限、加事务 | 代理 |
| 给复杂系统提供简单入口 | 外观 |
| 表示树形结构 | 组合 |
| 复用大量小对象 | 享元 |
| 多个处理器按顺序处理请求 | 责任链 |
| 把操作封装成对象 | 命令 |
| 解析简单语法/表达式 | 解释器 |
| 隐藏集合遍历细节 | 迭代器 |
| 多个对象通过协调者交互 | 中介者 |
| 保存状态用于撤销恢复 | 备忘录 |
| 一对多通知 | 观察者 |
| 对象状态不同导致行为不同 | 状态 |
| 多种算法可替换 | 策略 |
| 固定流程，部分步骤交给子类 | 模板方法 |
| 对稳定对象结构增加新操作 | 访问者 |

## 12. 学习记忆建议

1. 先记意图，不要先背类图。
2. 同一组模式放进同一个业务域里看代码差异。
3. 看到“包装一层”时，先问目的：转接口、加功能、控访问，还是拆维度。
4. 看到 `if-else` 很多时，判断它是在选算法、选状态，还是选处理人。
5. 看到对象创建复杂时，判断是选类型、组装步骤、复制模板，还是创建产品族。
6. 看到对象交互复杂时，判断是一对多通知，还是多对多协调。
