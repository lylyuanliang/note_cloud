### 外观模式教程

外观模式（Facade Pattern）是一种结构型设计模式。它为复杂子系统提供一个统一、简单的访问入口，让客户端不需要直接理解子系统内部细节。

### 1. 外观模式结构

- **Facade（外观）** ：对外提供简单接口，协调子系统。
- **Subsystem（子系统）** ：完成具体功能的多个类。
- **Client（客户端）** ：只依赖外观对象。

### 2. 外观模式的优缺点

#### 优点

* **简化调用** ：客户端只面对一个高层入口。
* **降低耦合** ：减少客户端和多个子系统类之间的直接依赖。
* **便于组织流程** ：可以把多步骤流程封装为一个方法。

#### 缺点

* **外观类可能过大** ：如果什么都放进去，会变成上帝类。
* **可能隐藏重要细节** ：封装不合理会让调用者难以处理异常和分支。

### 3. 适用场景

* 系统内部复杂，但对外只需要提供简单操作。
* 需要为多步骤流程提供统一入口。
* 需要隔离客户端和子系统变化。

### 4. Java 实现示例

```java
class Cpu {
    void start() {
        System.out.println("CPU启动");
    }
}

class Memory {
    void load() {
        System.out.println("内存加载");
    }
}

class Disk {
    void read() {
        System.out.println("磁盘读取");
    }
}

class ComputerFacade {
    private final Cpu cpu = new Cpu();
    private final Memory memory = new Memory();
    private final Disk disk = new Disk();

    public void start() {
        cpu.start();
        memory.load();
        disk.read();
        System.out.println("电脑启动完成");
    }
}

public class Client {
    public static void main(String[] args) {
        ComputerFacade computer = new ComputerFacade();
        computer.start();
    }
}
```

### 5. 注意事项

- 外观模式不是封装所有功能，而是封装常用的高层流程。
- 外观类不应该阻止客户端在必要时直接访问子系统。
- 外观模式常用于服务层、SDK 门面和复杂流程编排。
