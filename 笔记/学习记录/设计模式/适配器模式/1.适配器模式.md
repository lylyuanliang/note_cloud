### 适配器模式教程

适配器模式（Adapter Pattern）是一种结构型设计模式。它将一个类的接口转换成客户端期望的另一个接口，使原本接口不兼容的类可以一起工作。

### 1. 适配器模式结构

- **Target（目标接口）** ：客户端期望使用的接口。
- **Adaptee（被适配者）** ：已经存在但接口不兼容的类。
- **Adapter（适配器）** ：实现目标接口，并在内部调用被适配者。
- **Client（客户端）** ：面向目标接口编程。

### 2. 适配器模式的优缺点

#### 优点

* **复用已有类** ：不需要修改原有代码。
* **隔离转换逻辑** ：接口转换集中在适配器中，客户端更简单。
* **符合开闭原则** ：可以新增适配器支持新的被适配者。

#### 缺点

* **增加类数量** ：会增加额外的适配器类。
* **可能增加理解成本** ：过度使用会让系统出现大量中间层。

### 3. 适用场景

* 需要复用已有类，但它的接口和当前系统不一致。
* 对接第三方 SDK、历史系统或老接口。
* 希望统一多个类似但接口不同的实现。

### 4. Java 实现示例

下面以支付接口为例。系统期望使用 `PayService`，但已有第三方支付类 `ThirdPartyPay` 的方法名和参数不同。

```java
interface PayService {
    void pay(String orderId, int amount);
}

class ThirdPartyPay {
    public void createPayment(String tradeNo, int cents) {
        System.out.println("第三方支付，交易号：" + tradeNo + "，金额：" + cents);
    }
}

class ThirdPartyPayAdapter implements PayService {
    private final ThirdPartyPay thirdPartyPay;

    public ThirdPartyPayAdapter(ThirdPartyPay thirdPartyPay) {
        this.thirdPartyPay = thirdPartyPay;
    }

    @Override
    public void pay(String orderId, int amount) {
        thirdPartyPay.createPayment(orderId, amount);
    }
}

public class Client {
    public static void main(String[] args) {
        PayService payService = new ThirdPartyPayAdapter(new ThirdPartyPay());
        payService.pay("ORDER_1001", 5000);
    }
}
```

### 5. 注意事项

- 适配器重点是解决接口不兼容，不是给对象增加新功能。
- 如果可以直接修改原接口，通常不需要适配器。
- 适配器常用于系统边界，比如接入外部服务或兼容旧代码。
