### 策略模式教程

策略模式（Strategy Pattern）是一种行为型设计模式。它定义一组算法，将每个算法封装成独立对象，并让它们可以相互替换。上下文对象不直接写具体算法，而是把算法执行委托给当前策略。

一句话记忆：**策略模式解决“同一件事有多种做法，并且做法需要灵活替换”的问题。**

### 1. 策略模式结构

- **Strategy（策略接口）** ：定义算法的公共接口。
- **ConcreteStrategy（具体策略）** ：实现某一种具体算法。
- **Context（上下文）** ：持有策略对象，并把算法执行委托给策略。
- **Client（客户端）** ：根据业务场景选择具体策略。

### 2. 不使用策略模式的问题

假设订单折扣规则写在一个方法里，最直接的代码通常是 `if-else`：

```java
class OrderPriceService {
    public int calculate(String userType, int price) {
        if ("normal".equals(userType)) {
            return price;
        }
        if ("vip".equals(userType)) {
            return (int) (price * 0.8);
        }
        if ("svip".equals(userType)) {
            return (int) (price * 0.7);
        }
        throw new IllegalArgumentException("未知用户类型：" + userType);
    }
}
```

这段代码能工作，但有几个问题：

- 折扣算法都挤在同一个方法里，方法会越来越长。
- 新增一种折扣时必须修改已有方法，容易影响旧逻辑。
- 折扣算法难以单独测试和复用。
- 如果多个地方都要按用户类型计算折扣，条件分支会重复出现。

策略模式的目标不是消灭所有 `if-else`，而是把**算法本身**从条件分支里拆出来。至于“选择哪个策略”，可以交给工厂、配置、枚举或依赖注入容器处理。

### 3. 策略模式的优缺点

#### 优点

* **消除算法分支** ：用多态替代大量按类型选择算法的 `if-else`。
* **算法可替换** ：运行时可以切换不同策略。
* **符合开闭原则** ：新增策略通常只需要新增策略类。
* **便于单独测试** ：每个策略类可以独立验证输入输出。

#### 缺点

* **策略类数量增加** ：每种算法通常对应一个类。
* **客户端需要了解策略差异** ：需要知道何时选择哪种策略。
* **选择逻辑仍然存在** ：策略模式拆掉的是算法分支，不是凭空知道该用哪个策略。

### 4. 适用场景

* 有多种算法可以完成同一类任务，例如折扣、排序、支付费率、权限判断。
* 算法需要独立扩展或替换。
* 代码中存在大量按类型选择算法的条件分支。
* 希望把算法作为配置项，在运行时或启动时注入。

不适合的情况：

* 只有一两个很简单且长期不会变化的分支，引入策略类可能过度设计。
* 分支判断主要表达对象状态流转，此时更可能是状态模式。
* 变化点是固定流程中的某几个步骤，此时更可能是模板方法模式。

### 5. Java 实现示例

下面用订单折扣来演示策略模式。

#### 5.1 策略接口

```java
interface DiscountStrategy {
    int discount(int price);
}
```

#### 5.2 具体策略

```java
class NormalDiscount implements DiscountStrategy {
    public int discount(int price) {
        return price;
    }
}

class VipDiscount implements DiscountStrategy {
    public int discount(int price) {
        return (int) (price * 0.8);
    }
}

class SuperVipDiscount implements DiscountStrategy {
    public int discount(int price) {
        return (int) (price * 0.7);
    }
}
```

#### 5.3 上下文对象

```java
class OrderPriceService {
    private DiscountStrategy discountStrategy;

    public OrderPriceService(DiscountStrategy discountStrategy) {
        this.discountStrategy = discountStrategy;
    }

    public void setDiscountStrategy(DiscountStrategy discountStrategy) {
        this.discountStrategy = discountStrategy;
    }

    public int calculatePrice(int price) {
        return discountStrategy.discount(price);
    }
}
```

#### 5.4 客户端使用

```java
public class Client {
    public static void main(String[] args) {
        OrderPriceService service = new OrderPriceService(new VipDiscount());
        System.out.println(service.calculatePrice(100)); // 80

        service.setDiscountStrategy(new SuperVipDiscount());
        System.out.println(service.calculatePrice(100)); // 70
    }
}
```

这里 `OrderPriceService` 不关心 VIP 折扣怎么算，也不关心 SVIP 折扣怎么算。它只依赖 `DiscountStrategy` 抽象。

### 6. 策略如何选择

策略模式经常被误解为“用了策略后就没有判断了”。实际上，选择策略的逻辑仍然需要存在，只是不应该和算法细节混在一起。

#### 6.1 简单工厂选择策略

```java
class DiscountStrategyFactory {
    public static DiscountStrategy getStrategy(String userType) {
        if ("normal".equals(userType)) {
            return new NormalDiscount();
        }
        if ("vip".equals(userType)) {
            return new VipDiscount();
        }
        if ("svip".equals(userType)) {
            return new SuperVipDiscount();
        }
        throw new IllegalArgumentException("未知用户类型：" + userType);
    }
}
```

客户端：

```java
DiscountStrategy strategy = DiscountStrategyFactory.getStrategy("vip");
OrderPriceService service = new OrderPriceService(strategy);
System.out.println(service.calculatePrice(100));
```

这种方式适合策略数量少、变化不频繁的场景。

#### 6.2 Map 注册策略

策略较多时，可以用 `Map` 注册，减少工厂里的条件分支：

```java
import java.util.HashMap;
import java.util.Map;

class DiscountStrategyFactory {
    private static final Map<String, DiscountStrategy> STRATEGIES = new HashMap<>();

    static {
        STRATEGIES.put("normal", new NormalDiscount());
        STRATEGIES.put("vip", new VipDiscount());
        STRATEGIES.put("svip", new SuperVipDiscount());
    }

    public static DiscountStrategy getStrategy(String userType) {
        DiscountStrategy strategy = STRATEGIES.get(userType);
        if (strategy == null) {
            throw new IllegalArgumentException("未知用户类型：" + userType);
        }
        return strategy;
    }
}
```

实际项目中，如果使用 Spring，也可以把不同策略声明成 Bean，再通过名称、枚举、注解或配置选择对应策略。

### 7. 和相近模式的区别

| 模式         | 关注点                       | 典型特征                               |
| ------------ | ---------------------------- | -------------------------------------- |
| 策略模式     | 多种算法可替换               | 策略通常由外部选择，Context 只负责委托 |
| 状态模式     | 同一对象在不同状态下行为不同 | 状态通常由对象内部流程切换             |
| 模板方法模式 | 固定流程，部分步骤变化       | 父类定义流程骨架，子类实现变化步骤     |

示例判断：

- “普通用户、VIP、SVIP 折扣算法不同”适合策略模式。
- “订单未支付、已支付、已发货时可执行操作不同”适合状态模式。
- “导入数据都要读取、校验、保存，只是读取和保存方式不同”适合模板方法模式。

### 8. 注意事项

- 策略模式关注算法替换，不负责保存对象状态变化。
- 示例中的 `int price` 可以理解为“分”为单位的金额；真实金额计算建议使用 `BigDecimal` 或统一的金额类型，避免小数精度和截断问题。
- 策略类最好保持无状态；如果策略必须有状态，需要注意线程安全和复用问题。
- 策略选择逻辑可以放在工厂、配置、枚举或依赖注入容器中。
- 不要为了消灭一个简单分支而强行引入策略模式，只有算法会扩展、复用或独立测试时才更有价值。
