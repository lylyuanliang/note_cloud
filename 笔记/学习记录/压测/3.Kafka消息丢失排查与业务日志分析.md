# Kafka 消息丢失排查与业务日志分析

本文整理一次“业务表缺数据，看起来像 Kafka 丢消息”的排查经验。核心结论是：Kafka lag 为 0 不等于业务处理成功，必须把生产端、broker、consumer group、业务日志和 ack 语义拆开验证。

## 1. 背景

问题表象：

```text
压测后前端告警数量不对
hatsmis_alarm_mark_info 存在缺失记录
现象类似 Kafka 消息丢失
```

重点样本：

```text
Kafka key: imageKey_caiji3_AL2_1783951054439e_1
Topic: algorithmtext_imageKey
消费组: algorithmText-group
业务现象: hatsmis_alarm_mark_info 未生成该车次部分告警记录
相关车次: pass_train_idx = 1783951054439e
```

最终结论：

```text
不是 Kafka broker 丢消息，而是消息已被消费。
业务处理时查不到过车记录，随后 consumer 仍提交 offset，导致 Kafka 不再重放。
```

## 2. 排查原则

遇到“消息丢失”类问题，不要先改代码，先拆链路：

1. 生产端是否真的发送了目标 key。
2. Kafka broker 中是否存在目标消息。
3. 消息被分配到哪个 partition、offset。
4. 消费组当前是否提交到该 offset 之后。
5. 哪个消费者节点持有该 partition。
6. 消费者日志里是否收到目标 key。
7. 收到后业务处理是否成功。
8. 如果业务失败，offset 是否仍被提交。

关键判断：

```text
Kafka lag 只能说明 offset 是否追上 log end offset，不能证明业务数据库写入成功。
```

## 3. 确认生产端 key

脚本 key 生成规则：

```text
imageKey_ + 采集编码 + "_" + 设备编号 + "_" + 过车ID + "_" + 顺位
```

例如：

```text
imageKey_caiji3_AL2_1783951054439e_1
```

先确认脚本确实可能生成目标 key，再继续排查 Kafka 和业务。

## 4. Kafka Broker 中确认消息是否存在

### 4.1 查看 topic 状态

```bash
/usr/local/kafka/bin/kafka-topics.sh \
  --bootstrap-server 10.1.9.5:9092 \
  --describe \
  --topic algorithmtext_imageKey
```

重点看：

1. partition 数量。
2. leader 是否正常。
3. ISR 是否包含 leader。
4. replication factor。

`ReplicationFactor=1` 是可靠性风险，但不能直接证明本样本丢失。

### 4.2 精确查目标 key

可使用本地沉淀脚本：

```text
D:\file_save\workspace\工作内容\列车在站\压测\kafka_find_key.py
```

示例：

```bash
python kafka_find_key.py \
  --bootstrap 10.1.9.5:9092 \
  --topic algorithmtext_imageKey \
  --key imageKey_caiji3_AL2_1783951054439e_1 \
  --start "2026-07-13 21:50:00" \
  --end "2026-07-13 22:05:00"
```

本次查到：

```json
{"key":"imageKey_caiji3_AL2_1783951054439e_1","topic":"algorithmtext_imageKey","partition":4,"offset":517583,"timestamp":"2026-07-13 21:57:35.703","value_size":985}
```

这一步证明：

```text
Kafka broker 中存在目标消息。
```

如果输出 `NOT_FOUND`，继续确认：

1. 时间窗口是否正确。
2. bootstrap 是否连到正确 Kafka。
3. topic 是否正确。
4. key 是否完全一致。
5. 是否查错环境，例如 `9092` 与 `9094`。

## 5. 消费组排查

### 5.1 查看消费组成员和分区归属

```bash
/usr/local/kafka/bin/kafka-consumer-groups.sh \
  --bootstrap-server 10.1.9.5:9092 \
  --describe \
  --group algorithmText-group \
  --members \
  --verbose
```

本次发现：

```text
algorithmText-group 中同时存在 10.1.9.5 和 10.1.9.251 两台消费者节点。
```

示例分区归属：

```text
partition 0 -> /10.1.9.251
partition 1 -> /10.1.9.251
partition 2 -> /10.1.9.251
partition 3 -> /10.1.9.5
partition 4 -> /10.1.9.5
```

注意：这表示 consumer group 把 partition 分配给哪台消费者实例处理，不表示 Kafka 数据存储在那台机器。

### 5.2 查看 lag 与 committed offset

```bash
/usr/local/kafka/bin/kafka-consumer-groups.sh \
  --bootstrap-server 10.1.9.5:9092 \
  --describe \
  --group algorithmText-group
```

如果目标 key 所在 partition 的 committed offset 已经超过目标 offset，则 Kafka 不会再自动投递这条消息。

但这只能说明：

```text
offset 已提交
```

不能说明：

```text
业务写库成功
```

## 6. 多节点日志排查

不要只查一台机器。应根据 consumer group 的 partition 分配去查对应节点日志。

日志路径示例：

```text
10.1.9.5:
/data/logs/hatsmis-data_processing

10.1.9.251:
/data/logs/hatsmis-data_processing
```

搜索目标 key：

```bash
cd /data/logs/hatsmis-data_processing
grep -n -- 'imageKey_caiji3_AL2_1783951054439e_1' hatsmis-data_processing.log.2026-07-13.*.log
```

本次在 `10.1.9.251` 找到：

```text
收到算法检测 Kafka 消息
进入 AlgorithmHandler
处理算法检测异常
```

这一步证明：

```text
消息不是没有被消费，而是消费后业务处理失败。
```

## 7. 业务异常分析

展开目标 key 附近日志：

```bash
sed -n '46480,46535p' /data/logs/hatsmis-data_processing/hatsmis-data_processing.log.2026-07-13.13.log
```

关键链路：

```text
收到算法检测 Kafka 消息
处理算法检测结果
SELECT ... FROM hatsmis_htpm_pass_record WHERE pass_train_idx = ?
Parameters: 1783951054439e
Total: 0
查询过车记录：未找到
处理算法检测异常
ServiceException: 过车记录不存在
```

分析顺序：

1. 是否进入 consumer。
2. 是否找到 handler。
3. handler 查询了什么业务表。
4. SQL 参数是什么。
5. 查询结果是什么。
6. 异常发生在入库前还是入库后。

本次异常发生在告警入库前，因此 `hatsmis_alarm_mark_info` 缺失是合理结果。

## 8. 为什么业务失败后 Kafka 不重放

当 consumer 结构类似下面这样时：

```java
try {
    if (handler != null) {
        handler.handle(message, receivedKey, acknowledgment);
    }
} catch (Exception e) {
    log.error("消费消息异常 - Key: {}", receivedKey, e);
} finally {
    if (acknowledgment != null) {
        acknowledgment.acknowledge();
    }
}
```

并且 Kafka 配置为：

```text
enable-auto-commit = false
AckMode.MANUAL_IMMEDIATE
```

含义是：

1. 自动提交关闭。
2. 业务代码调用 `acknowledge()` 时立即提交 offset。
3. 即使 handler 抛异常，只要 finally 执行 ack，Kafka offset 仍会前进。

因此会出现：

```text
业务数据库缺数据，但 Kafka lag = 0。
```

这是 Kafka 消费确认语义和业务成功语义不一致。

## 9. advertised.listeners 配错导致跨环境混入

本次还有一个关键环境问题：`10.1.9.251` 上 Kafka 的 `advertised.listeners` 配成了 `10.1.9.5:9092`。

错误链路：

```text
9.251 data_processing
  -> bootstrap 连接 10.1.9.251:9092
  -> 从 metadata 拿到 advertised 地址 10.1.9.5:9092
  -> 后续实际连接 10.1.9.5:9092
  -> 加入 10.1.9.5 Kafka 上的 algorithmText-group
```

影响：

1. 9.251 消费者混入 9.5 的同名消费组。
2. 9.251 抢到 9.5 Kafka 的部分 partition。
3. 如果 9.251 写的是另一套数据库，9.5 对应数据库会缺业务结果。
4. Kafka lag 仍可能为 0，因为 offset 已由同一个 group 提交。

修复：

```text
advertised.listeners=PLAINTEXT://10.1.9.251:9092
```

修复后判断标准：

```text
10.1.9.5:9092 的 algorithmText-group 中不能出现 /10.1.9.251。
10.1.9.251:9092 的 algorithmText-group 中可以出现 /10.1.9.251，但这是 9.251 自己环境的消费组。
```

## 10. 部署一致性检查

多节点消费组必须确认部署版本一致，否则不同 partition 会被不同逻辑处理。

本次发现：

```text
10.1.9.251 data_processing jar 时间: 2026-07-03 14:14
10.1.9.5 data_processing jar 时间: 2026-07-14 17:49
```

影响：

1. 同一 consumer group 中，不同 partition 可能由不同版本代码处理。
2. 压测结果会混杂。
3. 无法稳定证明修复是否生效。

压测前应统一 jar 版本，并确认 Nacos、Redis、DB、Kafka bootstrap 成套隔离。

## 12. 治理建议

### 12.1 业务处理语义

算法消息依赖过车记录，存在天然乱序风险。不能把“过车记录暂时不存在”当作永久失败。

建议：

1. 对 `过车记录不存在` 做可恢复处理。
2. 采用延迟重试、待匹配表、Redis 暂存或补偿任务。
3. 只有业务成功入库后再提交 Kafka offset。
4. 如果为了避免阻塞 Kafka 消费线程，可以先落库到待处理表，再 ack Kafka，后续由补偿任务保证业务最终一致。

### 12.2 Kafka ack 语义

避免 consumer 层无条件 ack：

1. 可恢复异常不应直接 ack。
2. 不可恢复异常应记录死信或失败表，再 ack。
3. 业务成功、失败补偿落地、死信落地，这三种状态至少要有一个成立，才能 ack。

### 12.3 环境治理

1. Kafka bootstrap、数据库、Redis、Nacos 配置要成套隔离。
2. `advertised.listeners` 必须指向当前环境正确地址。
3. 如果本机 Kafka 不使用，建议停用或隔离。
4. 多节点 data_processing jar 必须统一。

## 13. 复盘要点

1. 不要因为业务表缺数据就直接认为 Kafka 丢消息。
2. 先证明 Kafka 中是否存在目标 key。
3. 根据 partition 和 consumer group 分配找正确机器日志。
4. `lag=0` 不等于业务成功。
5. consumer 无条件 ack 会掩盖业务失败。
6. 多节点部署版本不一致会干扰压测结论。
7. `advertised.listeners` 配错会让消费者加入错误环境的消费组，实质抢占 partition。
