# Kafka 诊断脚本通用用法

本文记录 Kafka 诊断脚本的通用用法，目标是在不同业务场景中复用“按 key 查消息”和“按业务 ID 扫描多 topic 消息”的能力。

## 1. 脚本能力概览

| 脚本 | 通用能力 | 适合场景 |
| --- | --- | --- |
| `kafka_find_key.py` | 按精确 Kafka key 跨所有 partition 查找消息 | 判断某条消息是否真的进入 Kafka broker |
| `kafka_scan_pass_messages.py` | 按业务 ID 在多个 topic 中扫描相关消息，并统计顺位缺失 | 判断某个业务对象在多个 topic 中的消息是否齐全 |

两个脚本的共同特性：

1. 只读 Kafka。
2. 不提交消费位点。
3. 不修改 Kafka 消息。
4. 不访问或修改数据库。
5. 使用时间窗口定位消息，避免全量扫描成本过高。

## 3. 按精确 key 查消息

### 3.1 使用场景

当业务表缺数据、页面数量不对、下游没有处理结果时，先确认 Kafka broker 中是否存在目标消息。

适合回答：

```text
这条 key 是否进入了 Kafka？
在哪个 partition？
offset 是多少？
Kafka 时间戳是多少？
value 大小是多少？
是否有多条同 key 消息？
```

### 3.2 命令模板

```powershell
python kafka_find_key.py `
  --bootstrap <host:port> `
  --topic <topic-name> `
  --key <exact-key> `
  --start "YYYY-MM-DD HH:mm:ss" `
  --end "YYYY-MM-DD HH:mm:ss"
```

示例：

```powershell
python kafka_find_key.py `
  --bootstrap 10.1.9.5:9092 `
  --topic algorithmtext_imageKey `
  --key imageKey_caiji3_AL2_1783951054439e_1 `
  --start "2026-07-13 21:50:00" `
  --end "2026-07-13 22:05:00"
```

### 3.3 常用参数

| 参数 | 说明 |
| --- | --- |
| `--bootstrap` | Kafka bootstrap servers |
| `--topic` | 要扫描的 topic |
| `--key` | 精确匹配的 Kafka message key |
| `--start` | 开始时间，支持本地时间字符串或 epoch 秒/毫秒 |
| `--end` | 结束时间，支持本地时间字符串或 epoch 秒/毫秒 |
| `--timezone` | 默认 `Asia/Shanghai` |
| `--all` | 返回全部匹配记录；默认找到第一条就返回 |
| `--max-scan` | 最多扫描记录数，`0` 表示不限 |
| `--batch-size` | 每次 poll 的最大记录数 |
| `--poll-timeout-ms` | poll 超时时间 |
| `--value-preview` | 输出 value 前 N 字节，便于确认内容 |
| `--verbose` | 输出每个 partition 扫描进度 |

### 3.4 输出解释

找到消息时输出 JSON 行：

```json
{
  "key": "目标key",
  "topic": "topic-name",
  "partition": 4,
  "offset": 517583,
  "timestamp": "2026-07-13 21:57:35.703",
  "timestamp_ms": 1783951055703,
  "value_size": 985
}
```

含义：

| 字段 | 说明 |
| --- | --- |
| `partition` | 消息所在分区 |
| `offset` | 消息在该分区中的 offset |
| `timestamp` | Kafka 记录时间戳 |
| `value_size` | 消息 value 字节数 |
| `value_preview` | 指定 `--value-preview` 后才输出 |

如果输出：

```text
NOT_FOUND
```

不要直接判定 Kafka 丢消息，先检查：

1. 时间窗口是否覆盖发送时间。
2. bootstrap 是否连到正确环境。
3. topic 是否正确。
4. key 是否完全一致。
5. 消息是否写到了另一个 Kafka 端口或集群。

## 4. 按业务 ID 扫描多 topic 消息

### 4.1 使用场景

`kafka_scan_pass_messages.py` 当前以 `pass_train_idx` 命名，但通用思路是：

```text
给定一个业务 ID，在多个 topic 的指定时间窗口内查找所有 key 或 value 中包含该 ID 的消息。
```

适合回答：

```text
某个业务对象相关消息是否都进了 Kafka？
哪些 topic 有消息？
每个 topic 有多少条？
消息分布在哪些 partition？
哪些顺位或序号缺失？
是否需要打开 value 预览确认内容？
```

### 4.2 命令模板

```powershell
python kafka_scan_pass_messages.py `
  --bootstrap <host:port> `
  --pass-id <business-id> `
  --topics <topic1> <topic2> <topic3> `
  --start "YYYY-MM-DD HH:mm:ss" `
  --end "YYYY-MM-DD HH:mm:ss"
```

示例：

```powershell
python kafka_scan_pass_messages.py `
  --bootstrap 10.1.9.5:9092 `
  --pass-id ZNHJ260717092815 `
  --topics collecttext_collectKey collecttext_imagePathKey `
  --start "2026-07-17 09:27:00" `
  --end "2026-07-17 09:30:00"
```

### 4.3 常用参数

| 参数 | 说明 |
| --- | --- |
| `--bootstrap` | Kafka bootstrap servers |
| `--pass-id` | 要匹配的业务 ID；脚本会在 key 和 value 中搜索 |
| `--start` | 开始时间 |
| `--end` | 结束时间 |
| `--topics` | 要扫描的 topic 列表 |
| `--timezone` | 默认 `Asia/Shanghai` |
| `--min-seq` | 顺位缺失检查下界 |
| `--max-seq` | 顺位缺失检查上界 |
| `--show-value` | 输出 value 前 N 个字符 |
| `--batch-size` | 每次 poll 的最大记录数 |
| `--poll-timeout-ms` | poll 超时时间 |

### 4.4 输出解释

脚本按 topic 输出：

```text
===== TOPIC collecttext_collectKey =====
FOUND: 29
SEQS_UNIQUE: [1, 2, 3, ...]
MISSING_1_29: []
PARTITION_COUNTS: {0: 8, 1: 7, 2: 14}
2026-07-17 09:28:15.123 p=1 o=123456 seq=1 key=...
```

字段含义：

| 输出 | 说明 |
| --- | --- |
| `FOUND` | 当前 topic 匹配消息数 |
| `SEQS_UNIQUE` | 从 key 末尾提取到的唯一顺位 |
| `MISSING_1_29` | 指定顺位范围内缺失的顺位 |
| `PARTITION_COUNTS` | 匹配消息在各 partition 的数量 |
| `p` | partition |
| `o` | offset |
| `seq` | 从 key 末尾解析出的顺位 |
| `value_preview` | 指定 `--show-value` 后才输出 |

## 5. 通用化改造点

`kafka_find_key.py` 已经基本通用，业务无关。

`kafka_scan_pass_messages.py` 有几个业务痕迹，复用到其他业务时建议改造：

| 当前字段 | 通用化建议 |
| --- | --- |
| `--pass-id` | 改名为 `--business-id` 或 `--contains` |
| `DEFAULT_TOPICS` | 改为不设置默认 topic，要求调用方显式传 `--topics` |
| `extract_seq` | 支持通过正则参数指定序号提取规则 |
| `--min-seq` / `--max-seq` | 保留为可选序号完整性检查 |
| 输出标题中的 `PASS_ID` | 改成 `BUSINESS_ID` |

通用扫描模型：

```text
scan Kafka topics in time range
  -> match if business_id in key or value
  -> print topic / partition / offset / timestamp / key
  -> optionally parse sequence
  -> optionally print value preview
```

## 6. 排查流程建议

### 6.1 单条消息缺失

1. 用业务日志或请求记录拿到 Kafka key。
2. 用 `kafka_find_key.py` 精确查 key。
3. 找到 partition 和 offset。
4. 查 consumer group committed offset 是否超过该 offset。
5. 查该 partition 当时分配给哪个消费者。
6. 去对应机器业务日志中查 key。
7. 判断是 Kafka 未写入、未消费、业务失败还是 ack 语义问题。

详细排查方法见：[Kafka 消息丢失排查与业务日志分析](./3.Kafka消息丢失排查与业务日志分析.md)。

### 6.2 某个业务对象结果不完整

1. 用业务 ID 限定时间窗口。
2. 用 `kafka_scan_pass_messages.py` 扫描相关 topic。
3. 看每个 topic 的 `FOUND` 数量。
4. 看 `MISSING` 是否有缺失顺位。
5. 用 `--show-value` 确认消息内容。
6. 再结合数据库计数判断是 Kafka 输入缺失还是业务处理缺失。

## 7. 注意事项

1. 时间窗口越小越好，避免扫描成本过高。
2. `--value-preview` 可能输出敏感字段，只在排查需要时打开。
3. `enable_auto_commit=False` 保证脚本不会提交消费位点。
4. 脚本使用 `assign + seek` 直接扫描分区，不加入业务消费组。
5. 查到 Kafka 中存在消息，只能证明 broker 收到了消息，不代表业务处理成功。
6. 查不到消息时，优先检查环境、topic、key、时间窗口，再判断是否真的未写入。

## 8. 可复用命令速查

精确查 key：

```powershell
python kafka_find_key.py --bootstrap 10.1.9.5:9092 --topic topic_a --key exact_key --start "2026-07-17 09:00:00" --end "2026-07-17 09:10:00"
```

查 key 并预览 value：

```powershell
python kafka_find_key.py --bootstrap 10.1.9.5:9092 --topic topic_a --key exact_key --start "2026-07-17 09:00:00" --end "2026-07-17 09:10:00" --value-preview 300
```

返回同 key 全部消息：

```powershell
python kafka_find_key.py --bootstrap 10.1.9.5:9092 --topic topic_a --key exact_key --start "2026-07-17 09:00:00" --end "2026-07-17 09:10:00" --all
```

按业务 ID 扫描多个 topic：

```powershell
python kafka_scan_pass_messages.py --bootstrap 10.1.9.5:9092 --pass-id BUSINESS-001 --topics topic_a topic_b topic_c --start "2026-07-17 09:00:00" --end "2026-07-17 09:10:00"
```

检查 1 到 60 的顺位缺失：

```powershell
python kafka_scan_pass_messages.py --bootstrap 10.1.9.5:9092 --pass-id BUSINESS-001 --topics topic_a topic_b --start "2026-07-17 09:00:00" --end "2026-07-17 09:10:00" --min-seq 1 --max-seq 60
```
