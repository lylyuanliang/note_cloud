# 算法告警重复投递与并发幂等设计复盘

本文整理算法告警消息在高并发和重复投递场景下的幂等优化经验。它重点回答：为什么“批内去重 + DB 查库去重”仍然会重复入库，以及为什么 Redis `SETNX` 能作为并发幂等闸门。

- [Kafka 异步业务链路压测经验总结](./1.Kafka异步业务链路压测经验总结.md)
- [Kafka 消息丢失排查与业务日志分析](./3.Kafka消息丢失排查与业务日志分析.md)

## 1. 问题结论

本次优化解决的是：算法告警消息在高并发或重复投递场景下，`hatsmis_alarm_mark_info` 明细可能重复入库，进而导致过车告警数、车辆告警数被放大的问题。

优化后，本地已通过至少 10 个过车 id 的压测场景：

| 场景 | 过车 id | 车辆 | 图片 | 告警明细 | Kafka ack | DB完成耗时 | 结果 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| 正常混合高并发 | 10 | 600 | 1800 | 5400 | 4220/4220 | 85.719s | PASS |
| 算法重复重放幂等 | 10 | 600 | 1800 | 5400 | 6020/6020 | 156.472s | PASS |

## 2. 现象

历史压测中，`duplicate-algorithm-replay` 场景会发送同一业务算法结果的重复消息。Kafka key 或 DTO key 会变化，但业务维度相同。

旧逻辑中服务层已有两层去重：

1. 批内去重：同一个 batch 内相同业务 key 只保留一条。
2. 查库去重：插入前查询 `hatsmis_alarm_mark_info` 中已有算法告警，再过滤已存在业务 key。

但压测中仍会出现：

```text
hatsmis_alarm_mark_info 明细多写
hatsmis_htpm_pass_record.alarm_count 被放大
车辆分表 alarm_count 被放大
```

这不是 Kafka ack 丢失，也不是汇总字段单独算错，而是告警明细重复入库后，汇总字段按错误明细回算。

## 3. 根因：查库再插入不是并发安全

旧逻辑依赖数据库已提交状态判断重复：

```text
线程 A 查库：不存在
线程 B 查库：不存在
线程 A insert
线程 B insert
```

当两个消费者线程同时处理同一业务告警时，都可能在对方事务提交前查询到“数据库不存在”，随后都执行 insert。

缺口在于：

```text
DB 查重只能看到已提交数据，无法覆盖事务提交前的并发窗口。
```

## 4. 业务唯一 key 不应依赖 Kafka key

重复重放场景中，Kafka key / DTO key 可以变化，但业务含义相同。因此幂等 key 要基于业务唯一维度构造，而不是基于消息传输层 key。

业务 key 核心字段包括：

1. `passTrainIdx`
2. `sequenceNumber`
3. `equipmentNumber`
4. `detectionItemCode`
5. 坐标与框尺寸：`xaxis/yaxis/width/height`
6. 图片定位：优先包含 `fileId`，也可包含 `imageUrl`

原则：

```text
同一个业务告警，无论 Kafka key、DTO key、投递次数如何变化，都应得到同一个业务 key。
```

## 5. 优化方案：Redis SETNX 并发幂等闸门

本次在 DB 去重后、真正 insert 前增加 Redis 原子幂等键：

```text
alarm_mark_info:algorithm:business_key:{业务唯一维度}
```

处理流程：

1. 保留原有批内去重。
2. 保留原有查库去重。
3. 对剩余待入库数据，按业务 key 执行 Redis `SETNX`。
4. 只有抢到全部业务 key 的数据才允许继续生成 idx 并 insert。
5. 抢不到 key 的数据视为并发重复消息，跳过 insert。
6. 如果 insert 返回 false 或抛异常，释放本次抢占的 Redis key，避免误拦后续重试。
7. 成功 insert 后保留 Redis key 的 TTL，短期重复消息不会再次入库。
8. 没有业务 key 的数据保持旧行为，不误跳过。

核心操作：

```java
Boolean acquired = stringRedisTemplate.opsForValue().setIfAbsent(
        redisKey,
        ALARM_MARK_INFO_BUSINESS_KEY_VALUE,
        CACHE_EXPIRE_HOURS,
        TimeUnit.HOURS);
```

`setIfAbsent` 是原子操作。并发线程处理同一业务告警时，只有一个线程能拿到 Redis key。

## 6. 失败释放逻辑很关键

如果只抢 Redis key，不处理失败释放，会产生新的问题：

```text
线程抢到 key
  -> insert 失败或抛异常
  -> key 仍保留
  -> 后续真实重试被误判为重复
```

因此必须在失败路径释放已抢占 key：

```java
try {
    // 生成 idx 并 insert
    Boolean inserted = alarmMarkInfoMapper.insertBatch(...);
    if (!Boolean.TRUE.equals(inserted)) {
        releaseRedisBusinessKeys(acquiredRedisKeys);
    }
    return inserted;
} catch (RuntimeException | Error e) {
    releaseRedisBusinessKeys(acquiredRedisKeys);
    throw e;
}
```

注意：成功 insert 后不释放 key，而是保留 TTL，用于屏蔽短期重复投递。

## 7. 为什么能生效

旧逻辑的问题是：

```text
重复判断发生在 DB 已提交视角，无法识别提交前并发。
```

新逻辑的关键变化是：

```text
重复判断提前到 insert 前，并使用 Redis 原子 SETNX 作为跨线程/跨节点闸门。
```

只要业务 key 相同，即使 Kafka key、DTO key 不同，也会被判定为同一业务告警。因此：

1. 重复算法消息不会重复写入 `hatsmis_alarm_mark_info`。
2. 过车和车辆告警汇总不会被重复明细放大。
3. 算法消息乱序、重试、重复投递时业务结果保持稳定。

## 8. 验证方式

### 8.1 单元测试

命令：

```powershell
mvn -pl yunda-module-hatsmis-data_processing -am -Dtest=AlarmMarkInfoServiceImplTest '-Dsurefire.failIfNoSpecifiedTests=false' test
```

结果：

```text
Tests run: 9, Failures: 0, Errors: 0, Skipped: 0
BUILD SUCCESS
```

覆盖点：

1. 相同业务 key 批内去重。
2. 数据库已存在业务 key 时不插入。
3. Redis `SETNX=false` 时不插入。
4. 同一告警同时存在 `fileId` 和 `imageUrl`，第二个 key 抢占失败时释放第一个 key。
5. `insertBatch=false` 时释放已抢占 key。
6. `insertBatch` 抛异常时释放已抢占 key。
7. 无业务 key 的数据保持原插入行为。

### 8.2 本地压测

报告见：[Kafka 异步业务链路压测经验总结](./1.Kafka异步业务链路压测经验总结.md)。

6. 重复算法重放场景按业务 key 分组，重复分组数为 0。

## 9. 风险和边界

1. Redis 幂等 key 有 TTL，适合短期重复消息和并发重放防护。
2. 超过 TTL 后，如果同一业务消息再次到达，仍依赖 DB 去重兜底。
3. 本次没有新增数据库唯一索引，因此不会引入历史脏数据导致的 DDL 风险。
4. 如果未来业务允许同一图片、同一坐标、同一检测项重复产生多条告警，需要重新定义业务唯一 key。
5. 当前单节点有效消费并发受 topic 分区数限制，即使 listener concurrency 配得更高，分区不增加也无法线性提升消费并发。

## 10. 长期方案

Redis SETNX 是并发闸门，适合低风险快速修复和短期重复投递防护。

长期可以考虑为算法告警业务唯一维度增加数据库唯一约束，但前提是：

1. 清理历史重复数据。
2. 明确业务唯一 key 不会误伤合法重复。
3. 评估 DDL 对大表的锁表、耗时和回滚风险。
4. 设计 insert 冲突处理策略。

## 11. 后续建议

1. 保留 `duplicate-algorithm-replay` 作为固定回归场景。
2. 后续阶梯压测同时记录 Kafka lag、MySQL 慢 SQL、Redis 慢日志和 JVM GC。
3. 如果生产需要更高吞吐，先评估 topic 分区数和 listener concurrency。
4. 对核心幂等逻辑保留单元测试和压测回归双重验证。
