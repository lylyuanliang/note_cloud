## 1. 压测目标

本轮压测的目标不是只看 Kafka 发送速度，而是验证完整业务链路：

```text
压测脚本生产 Kafka 消息
  -> data_processing Kafka listener 消费
  -> 业务处理车辆、图片、算法告警
  -> 写入车辆表、图片表、告警明细表
  -> 回算过车告警数和车辆告警数
  -> 严格校验 DB 结果是否等于预期
```

因此报告中必须同时记录：

1. Kafka ack 是否全量返回。
2. 车辆记录是否达到预期。
3. 图片记录是否达到预期。
4. 告警明细是否达到预期。
5. 过车告警汇总是否与明细一致。
6. 车辆告警汇总是否与明细一致。
7. DB 是否最终完成。

## 2. 典型压测场景

| 场景 | 目的 |
| --- | --- |
| `collect-heavy` | 仅采集车辆高并发，验证分布式锁、`vehicle_count`、`trainNos` 聚合 |
| `image-before-collect` | 图片先到、采集后到，验证图片与车辆补齐 |
| `algorithm-before-collect` | 算法告警先到、车辆后到，验证 afterCommit 回算车辆/过车告警 |
| `duplicate-algorithm-replay` | 同一算法结果重复投递且 DTO key 改变，验证业务去重和幂等 |
| `mixed-fast` | 采集、图片、算法混合零间隔，验证综合高并发链路 |

这些场景覆盖的不只是吞吐，还覆盖消息乱序、重复投递、汇总回算和最终一致性。

## 4. 重要口径：发送完成不等于业务完成

压测报告里至少有三个耗时：

| 耗时 | 含义 |
| --- | --- |
| 发送耗时 | 压测脚本把消息发送到 Kafka 并拿到 ack 的时间 |
| DB完成耗时 | 业务侧 DB 记录和汇总字段达到预期的时间 |
| 墙钟耗时 | 外层执行视角的总耗时 |

对于异步链路，核心指标应优先看 `DB完成耗时`。发送耗时只能说明生产端和 Kafka ack 快，不代表业务处理完成。

报告中可以继续计算：

```text
每个过车耗时 = DB完成耗时 / 过车 id 数
每辆车厢耗时 = DB完成耗时 / 车辆总数
每个告警耗时 = DB完成耗时 / 告警明细总数
```

## 5. 严格计数校验

压测脚本的观测逻辑不能只判断“大于等于预期”，应严格判断：

```text
实际值 == 预期值
```

因为业务问题可能有两类：

1. 少写：消息未处理、业务异常、乱序未补偿、offset 已提交。
2. 多写：重复投递、并发重复插入、幂等失效、汇总重复回算。

例如 `duplicate-algorithm-replay` 场景，如果只看“有数据”，很容易漏掉重复明细放大问题。必须同时校验：

```text
alarm_rows == expected_alarm_rows
pass_alarm_count == expected_alarm_rows
vehicle_alarm_sum == expected_alarm_rows
```

## 6. 并发能力判断

本地配置中：

```text
data_processing Kafka listener concurrency = 3
业务 topic 分区数 = 3
单节点单消费组有效消费并发 ≈ min(concurrency, partitions) = 3
压测脚本生产端 worker = 20
```

结论：

1. 生产端 worker 多，不代表消费端就能等比例并行。
2. Kafka 单消费组内，一个 topic partition 同一时刻只能被一个消费者实例处理。
3. 如果 topic 只有 3 个分区，单节点 listener concurrency 配到 10 也不能得到 10 倍消费并发。
4. 提升吞吐时要同时考虑 topic 分区数、listener concurrency、MySQL 批量写入、Redis 命令耗时和 JVM GC。

## 7. “已验证上限”不是理论极限

报告中的“已验证上限”只表示本轮跑过并通过的最大业务规模。

例如：

```text
10 个过车 id
600 辆车
1800 张图片
5400 条告警明细
Kafka ack 6020/6020
DB 完成耗时 156.472s
```

这不等于系统理论极限。要得到理论极限，需要继续做阶梯压测，例如：

```text
10x80
12x80
16x80
20x80
```

同时观察：

1. Kafka lag。
2. MySQL 慢 SQL。
3. Redis 慢日志。
4. JVM GC。
5. CPU、内存、磁盘 IO。
6. 业务错误日志。

```text

7. 结果文件列表
```

## 9. 后续建议

1. 保留 `duplicate-algorithm-replay` 作为固定回归场景。
2. 每轮报告同时记录 Kafka lag、MySQL 慢 SQL、Redis 慢日志和 JVM GC。
3. 对“少写”和“多写”都做严格计数。
4. 区分发送端吞吐和业务端最终完成耗时。
5. 如果需要提升理论上限，先评估 topic 分区数和 listener concurrency，再评估 DB 批量写入和索引策略。
6. 对 Kafka “消息丢失”类问题，按 [Kafka 消息丢失排查与业务日志分析](./3.Kafka消息丢失排查与业务日志分析.md) 的链路逐层验证。
