## 1. 前置说明 

### `@JsonTypeInfo`

Jackson 库中的注解之一，用于在序列化和反序列化 JSON 数据时处理多态性，特别是在处理继承结构时非常有用。作用于类/接口，被用来开启多态类型处理，对基类/接口和子类/实现类都有效

* `use`：指定用于序列化和反序列化的类型标识符：

  * `JsonTypeInfo.Id.CLASS`：使用类的完全限定名。
  * `JsonTypeInfo.Id.NAME`：使用自定义类型名，需要配合 `@JsonSubTypes` 注解。
  * `JsonTypeInfo.Id.MINIMAL_CLASS`：若基类和子类在同一包内，使用类名(忽略包名)作为识别码
  * `JsonTypeInfo.Id.NONE`：不包含任何类型信息。
  * `JsonTypeInfo.Id.CUSTOM`：使用自定义的类型解析器处理类型信息。由@JsonTypeIdResolver对应
* `include`：定义类型信息的包含位置, 指定如何将类型信息包含在 JSON 中。可以是以下常量之一：

  * `JsonTypeInfo.As.PROPERTY`：作为数据的兄弟属性
  * `JsonTypeInfo.As.EXISTING_PROPERTY`：作为POJO中已经存在的属性
  * `JsonTypeInfo.As.EXTERNAL_PROPERTY`：作为扩展属性
  * `JsonTypeInfo.As.WRAPPER_OBJECT`：作为一个包装的对象
  * `JsonTypeInfo.As.WRAPPER_ARRAY`：作为一个包装的数组
* `property`(可选): 指定识别码的属性名称, 此属性只有当:

  * `use`为 `JsonTypeInfo.Id.CLASS`（若不指定property则默认为 `@class`）、`JsonTypeInfo.Id.MINIMAL_CLASS`(若不指定 `property`则默认为 `@c`)、`JsonTypeInfo.Id.NAME`(若不指定 `property`默认为 `@type`)，
  * `include`为 `JsonTypeInfo.As.PROPERTY`、`JsonTypeInfo.As.EXISTING_PROPERTY`、`JsonTypeInfo.As.EXTERNAL_PROPERTY`时才有效
* `visible`：设置类型识别字段是否继续传递给反序列化器。默认为 `false`，意味着这个字段只用于类型判断, 不再继续赋值给对象属性。如果设置为 `true`，则可以同时把这个字段设置到对象属性中, 本例一般需要设置为true。
* `defaultImpl`：如果类型识别码不存在或者无效，可以使用该属性来指定反序列化时使用的默认类型

### `@JsonSubTypes`

作用于类/接口，用来列出给定类的子类，只有当子类类型无法被检测到时才会使用它,一般是配合@JsonTypeInfo在基类上使用，比如：

```java
@JsonTypeInfo(use = JsonTypeInfo.Id.NAME,include =   JsonTypeInfo.As.PROPERTY,property = "typeName")  
@JsonSubTypes({@JsonSubTypes.Type(value=Sub1.class,name = "sub1"),@JsonSubTypes.Type(value=Sub2.class,name = "sub2")})
```

@JsonSubTypes的值是一个@JsonSubTypes.Type[]数组，里面枚举了多态类型(value对应子类)和类型的标识符值(name对应@JsonTypeInfo中的property标识名称的值，此为可选值，若不指定需由@JsonTypeName在子类上指定)

### `@JsonTypeName`

作用于子类，用来为多态子类指定类型标识符的值，比如：

```java
@JsonTypeName(value = "sub1")
```

## 2. 参数接收多态实现

上一篇中使用了自定义参数解析器 `HandlerMethodArgumentResolver` 来完成请求参数类型多态。本篇换一种方式, 使用 Jackson 自带的多态反序列化能力来实现同样的效果。

这种方式的核心思想是:

1. 请求体中仍然传入一个 `type` 字段。
2. 在父类上使用 `@JsonTypeInfo` 指定根据 `type` 字段判断具体类型。
3. 使用 `@JsonSubTypes` 或 `@JsonTypeName` 声明 `type` 和子类之间的映射关系。
4. Controller 中继续使用普通的 `@RequestBody`, 不需要自定义参数解析器。

### 2.1 功能描述

仍然沿用上一篇中的场景, 前端有三个表单报文:

```json
{
  "type": "base",
  "commonParam1": "commonParam1",
  "commonParam2": "commonParam2"
}
```

```json
{
  "type": "param1",
  "commonParam1": "commonParam1",
  "commonParam2": "commonParam2",
  "param1Value": "param1Value",
  "param1Value1": "param1Value1"
}
```

```json
{
  "type": "param2",
  "commonParam1": "commonParam1",
  "commonParam2": "commonParam2",
  "param2Value": "param2Value",
  "param2Value2": "param2Value2"
}
```

这三个报文都提交到同一个接口:

```java
@PostMapping("/test")
public String test(@RequestBody ParamBase param) {
    return JSONObject.toJSONString(param);
}
```

期望效果是:

* `type = base` 时, 反序列化为 `ParamBase`
* `type = param1` 时, 反序列化为 `Param1`
* `type = param2` 时, 反序列化为 `Param2`

## 3. 实现

### 3.1 定义父类

在父类上添加 `@JsonTypeInfo` 和 `@JsonSubTypes`。

```java
@Data
@JsonTypeInfo(
        use = JsonTypeInfo.Id.NAME,
        include = JsonTypeInfo.As.EXISTING_PROPERTY,
        property = "type",
        visible = true,
        defaultImpl = ParamBase.class
)
@JsonSubTypes({
        @JsonSubTypes.Type(value = ParamBase.class, name = "base"),
        @JsonSubTypes.Type(value = Param1.class, name = "param1"),
        @JsonSubTypes.Type(value = Param2.class, name = "param2")
})
public class ParamBase {

    private String type;

    private String commonParam1;

    private String commonParam2;
}
```

说明:

* `use = JsonTypeInfo.Id.NAME`: 使用逻辑名称来匹配类型, 比如 `param1`、`param2`。
* `include = JsonTypeInfo.As.EXISTING_PROPERTY`: 表示类型识别字段就是对象中已经存在的属性, 这里就是 `type`。
* `property = "type"`: 指定使用请求体中的 `type` 字段作为类型识别字段。
* `visible = true`: 让 `type` 字段在反序列化后仍然可以设置到对象属性中。如果不设置为 `true`, Jackson 可能只把它当作类型识别字段使用, 不再赋值给 `type` 属性。
* `defaultImpl = ParamBase.class`: 当没有传 `type` 或没有匹配到子类时, 默认使用 `ParamBase`。

### 3.2 定义子类

```java
@Data
@EqualsAndHashCode(callSuper = true)
public class Param1 extends ParamBase {

    private String param1Value;

    private String param1Value1;
}
```

```java
@Data
@EqualsAndHashCode(callSuper = true)
public class Param2 extends ParamBase {

    private String param2Value;

    private String param2Value2;
}
```

这种写法中, 子类本身不需要再添加注解, 因为父类的 `@JsonSubTypes` 已经把子类和 `type` 的对应关系写清楚了。

### 3.3 定义接口

接口仍然使用普通的 `@RequestBody`。

```java
@RestController
@RequestMapping("/jsonTypeInfo")
public class JsonTypeInfoController {

    @PostMapping("/test")
    public String test(@RequestBody ParamBase param) {
        return JSONObject.toJSONString(param);
    }
}
```

这里不需要做任何 Spring MVC 的额外配置。Spring MVC 在处理 `@RequestBody` 时, 默认会使用 `HttpMessageConverter` 读取请求体。如果项目使用的是 Spring Boot Web, JSON 反序列化一般会交给 Jackson 处理, 所以 Jackson 会自动识别 `@JsonTypeInfo` 这些注解。

## 5. 使用 `@JsonTypeName` 改造

上面的写法把所有子类都集中写在父类的 `@JsonSubTypes` 中。如果子类比较多, 父类会越来越长。此时可以把类型名称放到子类上。

父类:

```java
@Data
@JsonTypeInfo(
        use = JsonTypeInfo.Id.NAME,
        include = JsonTypeInfo.As.EXISTING_PROPERTY,
        property = "type",
        visible = true,
        defaultImpl = ParamBase.class
)
@JsonSubTypes({
        @JsonSubTypes.Type(value = ParamBase.class),
        @JsonSubTypes.Type(value = Param1.class),
        @JsonSubTypes.Type(value = Param2.class)
})
@JsonTypeName("base")
public class ParamBase {

    private String type;

    private String commonParam1;

    private String commonParam2;
}
```

子类:

```java
@Data
@EqualsAndHashCode(callSuper = true)
@JsonTypeName("param1")
public class Param1 extends ParamBase {

    private String param1Value;

    private String param1Value1;
}
```

```java
@Data
@EqualsAndHashCode(callSuper = true)
@JsonTypeName("param2")
public class Param2 extends ParamBase {

    private String param2Value;

    private String param2Value2;
}
```

注意: `@JsonTypeName` 只是把类型名称从 `@JsonSubTypes.Type(name = "...")` 中拆出来, 并不代表完全不需要注册子类。常见做法仍然是在父类的 `@JsonSubTypes` 中列出子类, 只是不用再写 `name`。

## 6. 和自定义参数解析器方案对比

### 6.1 `@JsonTypeInfo` 方案

优点:

* 实现简单, 不需要自定义 `HandlerMethodArgumentResolver`。
* Controller 中仍然使用标准的 `@RequestBody`。
* 由 Jackson 负责类型选择和对象转换, 代码量少。
* 适合请求体本身就是 JSON, 且类型字段就在 JSON 内部的场景。

缺点:

* 类型映射关系需要写在父类或 Jackson 配置中。
* 逻辑和 Jackson 强绑定, 如果后续不是 JSON 请求体, 这套方案就不适用。
* 对 `type` 的异常处理主要由 Jackson 抛出异常, 如果想做非常个性化的错误提示, 需要额外处理。

### 6.2 自定义参数解析器方案

优点:

* 控制能力更强, 可以自己决定如何读取请求、如何匹配类型、如何处理异常。
* 类型映射可以通过扫描注解、配置文件、数据库等方式维护。
* 不局限于 Jackson, 也不局限于标准 `@RequestBody` 的处理流程。

缺点:

* 代码量更多, 需要自己实现参数解析和注册逻辑。
* 需要考虑 request body 重复读取、异常处理、参数校验等细节。
* 对普通业务场景来说可能偏重。

## 7. 注意事项

### 7.1 `include` 的选择

如果请求体中的 `type` 本来就是业务字段, 推荐使用:

```java
include = JsonTypeInfo.As.EXISTING_PROPERTY
```

如果 `type` 只是为了 Jackson 识别类型, 并不一定在 Java 对象中存在, 可以使用:

```java
include = JsonTypeInfo.As.PROPERTY
```

本例中 `type` 是业务字段, 所以使用 `EXISTING_PROPERTY` 更合适。

### 7.2 `visible = true`

如果希望反序列化后对象中的 `type` 字段也有值, 建议设置:

```java
visible = true
```

否则 Jackson 可能会消费掉这个字段, 只把它用于类型判断, 不再赋值给对象属性。

### 7.3 未知类型

如果请求中传入了未知的 `type`, 例如:

```json
{
  "type": "unknown",
  "commonParam1": "commonParam1"
}
```

如果配置了:

```java
defaultImpl = ParamBase.class
```

则可能会使用 `ParamBase` 兜底。如果没有配置默认类型, Jackson 通常会抛出类型识别失败的异常。

具体是否应该配置 `defaultImpl`, 要看业务要求:

* 如果未知类型应该直接报错, 不建议配置默认类型。
* 如果没有传 `type` 时可以按基础类型处理, 可以配置 `defaultImpl = ParamBase.class`。

### 7.4 参数校验

如果不同子类有不同的校验规则, 可以直接在子类字段上添加校验注解:

```java
@Data
@EqualsAndHashCode(callSuper = true)
public class Param1 extends ParamBase {

    @NotBlank
    private String param1Value;

    private String param1Value1;
}
```

接口参数上继续使用 `@Valid`:

```java
@PostMapping("/test")
public String test(@Valid @RequestBody ParamBase param) {
    return JSONObject.toJSONString(param);
}
```

Jackson 会先根据 `type` 反序列化出具体子类, 然后 Spring 再对实际对象进行参数校验。

## 8. 总结

如果只是想解决 JSON 请求体中根据 `type` 自动匹配不同参数类的问题, `@JsonTypeInfo` 是更轻量的方案:

```java
@PostMapping("/test")
public String test(@RequestBody ParamBase param) {
    return JSONObject.toJSONString(param);
}
```

只要在 `ParamBase` 上配置好 Jackson 的多态注解, Spring MVC 原本的 `@RequestBody` 流程就可以直接完成类型分发。

如果后续需要更灵活的类型扫描、非 JSON 请求处理、特殊异常包装、动态类型注册等能力, 再考虑上一篇中的自定义参数解析器方案。
